# Weapp 95+ 提升总计划

日期：2026-03-27

## 1. 目标

本计划用于把当前小程序从“基础设施较强，但页面贯通与交付质量不稳定”的状态，提升到可稳定达到 95 分以上的产品与工程质量水平。

目标不是做一次零散修 bug，而是完成一轮覆盖五端的系统升级：

1. 消费侧
2. 商户侧
3. 运营商侧
4. 平台侧
5. 骑手侧

执行拆分：

- [weapp/docs/weapp_95_task_cards_20260327/README.md](weapp/docs/weapp_95_task_cards_20260327/README.md)

## 2. 当前判断

基于 2026-03-27 的审计结论，当前 weapp 的主要问题不是底层框架崩坏，而是页面层存在反复出现的交付断层：

1. 页面动作存在“看起来成功，但后端状态没变”的伪完成。
2. 页面跳转存在死链或注册缺失。
3. 列表过滤、分页和真实数据合同没有完全统一。
4. 冷启动、弱网、实时订阅等场景的页面收口不完整。
5. 高流量页面存在首屏请求扇出和渐进加载策略不稳定的问题。

结论：当前项目具备升级到 95+ 的基础，但必须按“整体验收标准”治理，而不是继续逐页零散补洞。

## 3. 95 分评分模型

95 分以上的目标，不以视觉精致度为核心，而以完整性、稳定性、性能、可维护性为核心。

### 3.1 评分维度

| 维度 | 当前判断 | 目标 |
| --- | --- | --- |
| 页面与服务贯通完整性 | 60-65 | 95+ |
| 冷启动与弱网稳定性 | 58-65 | 95+ |
| 首屏与长列表性能 | 58-70 | 95+ |
| 权限、路由、状态一致性 | 60-68 | 95+ |
| 五端交互反馈完备性 | 62-70 | 95+ |
| 工程复用与可维护性 | 75-80 | 95+ |

### 3.2 95 分达标标准

达到 95 分以上时，必须同时满足：

1. 不存在死链接、伪操作、只改本地状态不改后端状态的业务动作。
2. 五端核心链路全部具备 loading、success、empty、error、retry 完整状态。
3. 冷启动、前后台切换、弱网重试、token 刷新、实时订阅恢复都可稳定工作。
4. 关键首页和长列表不再存在明显的 per-item 首屏请求放大。
5. 路由、权限、状态、分页和过滤统一以真实合同为准，不靠页面层临时补丁维持。

## 4. 总体治理原则

### 4.1 全路径交付原则

每个功能一律按以下链路验收：

1. 路由可达
2. API 存在且参数正确
3. 页面状态完整
4. 事件处理闭环
5. 视图反馈完整
6. 刷新与回流策略明确

### 4.2 单一事实来源原则

以下信息不得由页面临时推断替代真实合同：

1. 权限
2. 订单类型过滤
3. 分页是否还有更多
4. 商户营业状态
5. 支付结果
6. 预订、桌台、配送、索赔等业务状态

### 4.3 首屏预算原则

高频入口页必须有明确首屏预算：

1. 首屏请求数量受控
2. 批量接口优先
3. 渐进补全优先于同步扇出
4. 已有虚拟列表、缓存、骨架能力优先复用

### 4.4 弱网优先原则

所有关键页都必须视弱网为默认环境而不是异常环境：

1. 失败可重试
2. 错误态可见
3. 空态与错态区分清楚
4. 不因一次失败把页面留在半完成状态

## 5. 跨端共性提升项

这些是五端共同必须先做的底座任务。

### 5.1 路由与页面注册治理

目标：清除所有死链与未注册页面跳转。

任务：

1. 建立页面路由清单，来源以 `app.json` 和分包配置为准。
2. 对所有 `navigateTo`、`redirectTo`、`reLaunch`、`switchTab` 做全量反查。
3. 将“跳转目标必须已注册”写入页面交付检查项。

验收：

1. 全量跳转无死链。
2. 五端首页和详情页互跳路径全部可达。

### 5.2 页面动作闭环治理

目标：清除所有伪操作与只改本地状态的业务动作。

任务：

1. 建立按钮到 API 的映射表。
2. 排查所有业务性 `setData` 切换是否对应真实服务调用。
3. 对开关、按钮、确认态、结果态统一补上成功与失败回流。

验收：

1. 无“toast 成功但后端未变”动作。
2. 所有关键动作失败时页面可恢复。

### 5.3 列表、筛选与分页治理

目标：把分页与过滤统一到真实合同。

任务：

1. 识别所有客户端分页后再过滤的页面。
2. 逐步把 `status`、`order_type`、角色、区域等条件下推到服务层。
3. 把 `hasMore` 的判定改成和后端结果一致。

验收：

1. 列表条数、空态、分页一致。
2. 不再出现“第一页为空但实际后面有数据”的情况。

### 5.4 实时订阅治理

目标：统一实时页面的冷启动、重连、前后台恢复策略。

任务：

1. 为骑手大厅、商户工作台、通知页建立统一订阅模板。
2. 明确何时连接、何时解绑、何时重连、何时回放消息。
3. 把“异步身份完成后必须补建订阅”写成标准模式。

验收：

1. 冷启动在线状态可直接收到实时事件。
2. 前后台切换后订阅自动恢复。

### 5.5 首屏性能治理

目标：控制入口页首屏请求数与节点数。

任务：

1. 为消费首页、骑手大厅、商户首页设首屏请求预算。
2. 把 per-item 补充信息改成批量接口或延迟水合。
3. 在 100+ 项列表中优先使用已有虚拟列表组件。

验收：

1. 高流量页面首屏请求数明显下降。
2. 低配机型滚动和切换不卡顿。

## 11. 分阶段实施计划

### Phase 0：治理基线

目标：先把规范和检查机制补齐。

任务：

1. 更新 weapp 指令，明确死链、伪操作、分页过滤、冷启动实时等反模式。
2. 建立五端页面清单与能力矩阵。
3. 建立“按钮到服务”“页面到路由”“列表到合同”的检查表。

### Phase 1：P0 止血项

目标：先消灭直接影响业务正确性的缺陷。

任务：

1. 修复骑手首页冷启动实时链路。
2. 修复堂食扫码死链。
3. 修复商户营业状态伪切换。
4. 修复商户预订订单分页过滤错位。

### Phase 2：高频页性能与状态完备性

目标：让消费首页、骑手大厅、商户首页达到高可用状态。

任务：

1. 重构外卖首页数据装配。
2. 补齐搜索页、列表页、详情页的 error 和 retry。
3. 优先接入已有虚拟列表能力到合适页面。

### Phase 3：五端能力补齐

目标：按能力矩阵补齐缺口页和缺失动作。

任务：

1. 商户侧补 KDS、投诉、员工、统计、设置域。
2. 运营商侧补审核、财务、统计的一致性。
3. 平台侧补高风险动作确认和全局视图统一。
4. 骑手侧补异常链路和钱包页状态完备性。

### Phase 4：95 分验收

目标：以量化标准做收口，而不是主观感受。

任务：

1. 每端重新打分。
2. 对五端关键链路做真机与弱网回归。
3. 跑 `npm run lint`、`npm run compile`、`npm run quality:check`。
4. 形成最终评分记录与残余风险表。

## 12. 95 分验收清单

所有端都需要通过以下检查：

1. 没有死链。
2. 没有伪操作。
3. 没有分页后再业务过滤导致的列表失真。
4. 核心实时页在冷启动在线状态下可直接工作。
5. 关键首页首屏请求预算达标。
6. loading、success、empty、error、retry 五态完整。
7. 权限和状态表达与后端一致。
8. 真机弱网下关键操作可恢复。

## 13. 推荐执行顺序

1. 先完成 Phase 0 和 Phase 1，把最危险的问题止血。
2. 再做消费首页、骑手大厅、商户首页三大高频入口的性能与状态治理。
3. 然后按能力矩阵补齐商户、运营商、平台、骑手的缺口页。
4. 最后做一次五端统一评分，目标是全部达到 95 分以上。

## 14. 最终结论

这次提升计划不应该被理解成“一轮普通 UI 重构”，而应该被理解成一次五端工程质量升级。

只有当以下三件事同时发生，weapp 才能稳定达到 95 分以上：

1. 页面层不再产生死链、伪操作和分页过滤错位。
2. 核心高频页具备弱网、冷启动、实时恢复、重试的完整韧性。
3. 五端都按同一套完整性交付标准建设，而不是各自为政。
