# CARD-13 骑手首页实时区与状态语义重构

状态：未开始

优先级：P1

所属批次：Batch 5

## 问题目标

在既有主链 contract 和弱网止血项基础上，把骑手首页收口成真正可信的实时工作台，明确区分无单、失败、订阅未恢复、定位异常等不同状态。

## 影响范围

- [weapp/miniprogram/pages/rider/dashboard/index.ts](weapp/miniprogram/pages/rider/dashboard/index.ts)
- [weapp/miniprogram/pages/rider/dashboard/index.wxml](weapp/miniprogram/pages/rider/dashboard/index.wxml)

## 当前前提

- 骑手侧已有任务卡已补主链 contract、弱网回归和部分错误壳，不重复追踪这些已收口项。
- 本卡聚焦实时工作台的信息架构、状态语义和观感统一升级。

## 已知问题

- 大厅、任务、概览、定位和导航入口堆在一个超大页面里，主任务不够聚焦。
- 子区块失败容易被伪装成“暂未展示”之类模糊空态。
- 对骑手这种高频实时页，状态语义一旦模糊，就会直接伤害抢单判断和任务信任感。

## 任务内容

- [ ] 重梳骑手首页的首屏结构，区分实时抢单区、当前任务区、辅助信息区。
- [ ] 区分真实空态、加载失败、订阅失败、定位失败、当前离线等用户可感知状态。
- [ ] 收口首页中的说明文案和状态文案，避免泛化空态吞掉真实错误原因。
- [ ] 检查实时区的按钮、状态卡和风险信息层级，让用户优先看到当前最重要的动作。

## 完成定义

- [ ] 骑手能快速判断当前是没单、加载失败还是订阅未恢复。
- [ ] 抢单区和当前任务区主次清楚，不再像多个页面拼在一个工作台里。
- [ ] 首页整体观感和状态承接明显更像实时工作台，而不是信息堆叠页。

## 验证要求

- [ ] 人工验证冷启动、有单、无单、订阅异常、定位异常、刷新失败六类场景。
- [ ] review 时使用 `.github/standards/weapp/REVIEW_CHECKLIST.md`，重点检查状态恢复、反馈语义和高频页主任务清晰度。
