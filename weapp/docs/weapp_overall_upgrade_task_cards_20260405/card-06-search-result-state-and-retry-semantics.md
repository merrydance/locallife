# CARD-06 外卖搜索结果态、建议态与失败恢复收口

状态：未开始

优先级：P1

所属批次：Batch 2

## 问题目标

让外卖搜索页明确区分建议态、结果态、空结果和失败恢复，不再把失败伪装成空，也不再让搜索页的语义在高频使用中持续漂移。

## 影响范围

- [weapp/miniprogram/pages/takeout/search/index.ts](weapp/miniprogram/pages/takeout/search/index.ts)
- [weapp/miniprogram/pages/takeout/search/index.wxml](weapp/miniprogram/pages/takeout/search/index.wxml)

## 已知问题

- 搜索建议失败会被直接降级成“无建议”一类空态。
- 搜索结果、建议列表和失败恢复之间的边界不够清楚。
- 作为高频入口，这类“小失败被伪装成空”的问题会持续伤害整端统一感和可信度。

## 任务内容

- [ ] 重新梳理搜索页的状态集合，明确初始态、输入中、建议成功、建议失败、结果成功、结果失败、空结果等语义。
- [ ] 区分“没有搜索建议”“建议请求失败”“没有搜索结果”“搜索结果失败”这四类用户可感知状态。
- [ ] 为建议失败和结果失败提供合理的页内恢复方式，而不是只用 toast 或直接回落成空。
- [ ] 优化搜索页的文案、说明和 CTA，使其保持高频页应有的直接性，不用说明堆叠掩盖状态不清。
- [ ] 检查搜索结果页的列表负担和刷新边界，避免每次重新输入都触发不必要的过量重查。

## 完成定义

- [ ] 用户能明确分清空结果和失败结果。
- [ ] 搜索建议与结果态之间切换自然，不会出现“其实失败了但看起来像没数据”。
- [ ] 搜索页在高频输入和弱网场景下仍保持统一、直接、可恢复。

## 验证要求

- [ ] 人工验证建议失败、结果失败、空结果、有结果四类场景。
- [ ] 验证输入切换、连续搜索和回退后再次进入的状态表现。
- [ ] review 时使用 `.github/standards/weapp/REVIEW_CHECKLIST.md`，重点检查反馈与提示、状态恢复和高频页统一性。
