# CARD-15 运营首页首屏优先级与待办工作台收口

状态：未开始

优先级：P1

所属批次：Batch 6

## 问题目标

在既有统计真值修复基础上，把运营首页首屏收口成真正的待办与决策工作台，而不是模块全塞首屏的聚合页。

## 影响范围

- [weapp/miniprogram/pages/operator/dashboard/index.ts](weapp/miniprogram/pages/operator/dashboard/index.ts)
- [weapp/miniprogram/pages/operator/dashboard/index.wxml](weapp/miniprogram/pages/operator/dashboard/index.wxml)

## 当前前提

- dashboard 与 analytics 的基础统计真值、部分入口分流和孤儿页清理已在既有 operator 任务卡中处理。
- 本卡聚焦首页的信息架构、首屏优先级和待办工作台表达，不重复追踪已接受的基础真值修复。

## 已知问题

- 首页首屏模块过多，待办、异常、排行、业务矩阵同时争抢注意力。
- 首页虽然信息很多，但不够清楚地表达“现在最该处理什么”。
- 对运营工作台来说，首屏过载会直接伤害弱网体验和决策效率。

## 任务内容

- [ ] 重梳首页首屏结构，只保留真正的高频待办、异常和关键决策信息。
- [ ] 把补充性统计、排行和次级信息后置，避免首页继续承担分析页职责。
- [ ] 统一首页卡片、状态区、入口按钮和说明文案的视觉节奏，让运营用户一眼能分清主次。
- [ ] 检查弱网和刷新失败时的首页承接方式，避免再次回落成“模块全有，但都不清楚”的混乱壳。

## 完成定义

- [ ] 运营首页首屏能快速回答“现在最该处理什么”。
- [ ] 首屏不再以模块数量制造“信息很全”的假繁忙感。
- [ ] 首页弱网和重入体验更稳，信息层级更清楚。

## 验证要求

- [ ] 人工验证首屏加载、刷新失败、从子页返回首页、弱网重进四类场景。
- [ ] review 时使用 `.github/standards/weapp/REVIEW_CHECKLIST.md`，重点检查视觉层级、主任务清晰度和性能预算。
