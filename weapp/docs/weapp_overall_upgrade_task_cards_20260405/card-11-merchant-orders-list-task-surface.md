# CARD-11 商户订单列表任务化与动作层级收口

状态：未开始

优先级：P1

所属批次：Batch 4

## 问题目标

把商户订单列表从“有动作的列表页”提升为真正的任务页，让风险单、超时单、待处理单和普通单的优先级清晰可见，动作层级稳定统一。

## 影响范围

- [weapp/miniprogram/pages/merchant/orders/list/index.ts](weapp/miniprogram/pages/merchant/orders/list/index.ts)
- [weapp/miniprogram/pages/merchant/orders/list/index.wxml](weapp/miniprogram/pages/merchant/orders/list/index.wxml)

## 已知问题

- 订单列表已承接主链路，但仍偏“列表展示”，不是明确的经营任务页。
- 高优先级单、风险单、超时单的视觉信号和动作优先级不够稳定。
- 主次按钮、拒单/处理等关键动作容易被列表密度和弱强调样式稀释。

## 任务内容

- [ ] 重梳订单列表首屏任务结构，明确待处理任务、普通单和补充信息的层级。
- [ ] 收口风险单、超时单、关键状态和下一步动作的视觉信号，避免用户需要逐行阅读才能判断优先级。
- [ ] 优化列表卡片的动作层级，让主动作、次动作和危险动作都有清晰的强调和承接方式。
- [ ] 检查局部骨架、失败重试、分页与筛选，避免运营高频页仍停留在“能拉出来列表就算完成”的阶段。

## 完成定义

- [ ] 订单列表成为明确的经营任务页，而不是普通信息列表。
- [ ] 用户能快速识别当前最需要处理的订单和对应动作。
- [ ] 高风险或高优先级订单有稳定、一致的视觉信号和操作路径。

## 验证要求

- [ ] 人工验证待处理订单、高优先级订单、风险单、分页继续加载、刷新失败五类场景。
- [ ] 验证主动作与危险动作在列表中的可见性和可点击性。
- [ ] review 时使用 `.github/standards/weapp/REVIEW_CHECKLIST.md`，重点检查主任务清晰度、动作层级和列表高频使用体验。
