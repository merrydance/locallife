# CARD-07 主体申请页信息架构与分段任务流重构

状态：未开始

优先级：P1

所属批次：Batch 3

## 问题目标

在当前已完成基础闭环修复的前提下，把主体申请页从超长滚动链收口成更清晰的分段任务流，让资料录入、位置确认、证照上传和提交结果承接不再彼此抢注意力。

## 影响范围

- [weapp/miniprogram/pages/merchant/settings/application/index.ts](weapp/miniprogram/pages/merchant/settings/application/index.ts)
- [weapp/miniprogram/pages/merchant/settings/application/index.wxml](weapp/miniprogram/pages/merchant/settings/application/index.wxml)

## 当前前提

- 主体申请页提交结果承接、状态文案、`onShow` 重拉门槛、OCR 提示等基础止血项已在既有 merchant 配置中心任务中接受。
- 本卡不重复追踪已接受的止血项，而是继续做信息架构、任务分段和观感统一升级。

## 已知问题

- 基础信息、经营位置、多个证照上传和提交动作仍然堆在一条长滚动链里。
- 页面内多个“处理中/待确认/待补资料”语义分布在长页面不同区块，用户不容易形成清晰任务感。
- 即便 contract 已较稳定，页面结构仍超过单页可读性阈值。

## 任务内容

- [ ] 重梳主体申请页的信息架构，明确资料录入、位置确认、证照上传、提交与结果承接几个任务段。
- [ ] 收口长页面中的说明文案、卡片组织和按钮布局，避免说明堆叠与重复解释。
- [ ] 检查 OCR 回填、定位回写、上传错误和保存失败在新结构中的承接方式，确保不因为分段而丢恢复路径。
- [ ] 让页面主动作在不同状态下保持明确，不再让多个区块各自制造“看起来都很重要”的竞争。

## 完成定义

- [ ] 用户能一眼看出当前在完成哪一段任务。
- [ ] 页面不再依赖超长滚动链来承载所有关键信息和动作。
- [ ] 资料流的说明、卡片、间距和按钮布局明显更统一、更易扫读。

## 验证要求

- [ ] 人工验证首次填写、OCR 回填、资料不完整、提交失败、返回重进五类场景。
- [ ] review 时使用 `.github/standards/weapp/REVIEW_CHECKLIST.md`，重点检查视觉、交互和状态恢复。
