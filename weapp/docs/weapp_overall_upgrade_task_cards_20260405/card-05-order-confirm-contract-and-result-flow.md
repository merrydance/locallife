# CARD-05 外卖确认页 contract、试算与支付承接真实化

状态：未开始

优先级：P0

所属批次：Batch 2

## 问题目标

把外卖确认页从“按商户线性放大的重页面 + 支付承接不可信”收口成 contract 真实、试算清楚、支付结果可信的结算页。

## 影响范围

- [weapp/miniprogram/pages/takeout/order-confirm/index.ts](weapp/miniprogram/pages/takeout/order-confirm/index.ts)
- [weapp/miniprogram/pages/takeout/order-confirm/index.wxml](weapp/miniprogram/pages/takeout/order-confirm/index.wxml)
- 相关购物车详情、配送费试算、下单创建、支付创建与结果回查逻辑

## 已知问题

- 页面初始化、价格试算和下单创建会随商户数线性放大。
- 支付承接中仍有占位式分支，说明高风险结果没有被当成完整状态面处理。
- 结算页既重又不够可信，会同时伤害速度感和支付信任感。

## 任务内容

- [ ] 核对购物车详情、配送费试算、下单创建和支付创建的真实 contract。
- [ ] 梳理确认页中哪些信息必须同步可见，哪些可以在用户确认前后渐进回读。
- [ ] 收口多商户场景下的串行调用和重复试算，避免无控制地线性放大。
- [ ] 清理“支付功能开发中”之类占位承接，把支付创建失败、未知结果、取消支付和已创建订单后的结果处理做成真实分支。
- [ ] 优化确认页的信息架构，让地址、商品、费用、支付相关动作有清晰层级，不再让重计算流程直接压垮首屏观感。

## 完成定义

- [ ] 结算页不存在占位式支付承接或结果伪装。
- [ ] 多商户场景下的初始化、试算和下单负担明显下降。
- [ ] 用户在弱网、回前台、支付取消和重复进入后仍能回到可信状态。

## 验证要求

- [ ] 人工验证正常下单、试算失败、支付创建失败、取消支付、弱网回前台五类场景。
- [ ] 对比改造前后的试算和下单请求放大情况。
- [ ] review 时使用 `.github/standards/weapp/REVIEW_CHECKLIST.md`，重点检查契约闭环、反馈与提示、性能与稳定性。
