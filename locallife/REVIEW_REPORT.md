# 后端总体审查报告（架构 -> 接口 -> 数据链路）

> 审查范围：后端服务入口、路由与中间件、核心业务模块（用户/商户/订单/配送/支付/风控/统计等），以及 handler -> store -> sqlc 的调用链。

## 1. 项目用途说明
本项目是一个“本地生活服务平台”后端：
- 支持用户、小程序登录与会话管理。
- 商户入驻与管理、菜品/套餐/库存/桌台/包间管理。
- 多类型订单（外卖/堂食/打包/预订）与支付、退款、配送。
- 骑手管理、运营商管理与区域管理。
- 会员营销（充值、余额、券/折扣）、推荐系统与行为埋点。
- 通知系统（含 WebSocket 推送）、风控与信用分体系。

## 2. 业务链路流程梳理（状态机）
以下为核心业务状态机的简化梳理：

### 2.1 订单状态机（外卖/堂食/打包/预订）
- 状态：`pending` → `paid` → `preparing` → `ready` → `courier_accepted` → `picked` → `delivering` → `rider_delivered` → `user_delivered` → `completed`
- 取消：任意阶段可进入 `cancelled`（受规则限制）
- 触发点：
  - 支付回调：更新支付单并异步处理订单状态推进。
  - 商户/骑手操作：接单、出餐、取餐、送达。
  - 自动/超时策略：如超时确认送达。

### 2.2 预订状态机
- 状态：`pending` → `paid` → `confirmed` → `checked_in` → `completed`
- 异常：`cancelled` / `expired` / `no_show`
- 触发点：支付（押金/全款）、商户确认、用户到店签到、超时未到店。

### 2.3 配送状态机
- 状态：`assigned`/`picking` → `picked` → `delivering` → `delivered` → `completed`
- 触发点：抢单、取餐确认、送达确认、围栏驻留自动推进（地理围栏）。

### 2.4 商户入驻状态机（新版）
- 草稿 `draft` → 提交 `submitted` → 审核 `approved` / `rejected` → 重置 `draft`
- 进件：`approved` → `pending_bindbank` → `bindbank_submitted` → `active`

### 2.5 骑手入驻状态机
- 草稿 `draft` → `submitted` → `approved` / `rejected` → `pending_bindbank` → `active`

### 2.6 运营商入驻状态机
- 草稿 `draft` → 提交 `submitted` → `approved` / `rejected`

## 3. 项目生产健壮性
**已有保障**
- 连接池与优雅关闭：数据库连接池参数、健康检查、Graceful Shutdown。
- 请求保护：统一超时中间件、速率限制、安全头、HSTS、CORS。
- 支付回调安全：签名验签、幂等通知记录、金额校验。
- 异步与告警：使用任务队列处理支付后续流程，失败触发告警。
- 资源访问安全：上传文件签名访问、敏感图片鉴权。

**潜在风险/改进点**
- 部分外部依赖（地图/微信）失败时仅返回 5xx，缺少降级重试与熔断。
- Webhook 幂等基于通知ID记录；若记录失败缺乏补偿机制与专用告警指标。
- Redis 关闭时核心功能（队列/缓存/推送）可退化，但缺少功能开关的统一可观测性说明。
- 部分 API 缺少一致的响应 envelope 或错误码策略（虽有中间件支持）。

## 4. 项目功能完成度
**已覆盖主要业务域：**
- 用户、商户、订单、配送、支付、评价、通知、风控、统计、推荐。
- 多角色权限：管理员/运营商/商户/骑手。

**完成度结论：**
- 主链路（订单-支付-配送-通知）完整。
- 运营商与商户入驻链路完整（含 OCR、进件、权限校验）。
- 会员营销（充值规则、代金券、折扣）与推荐系统已具备可用闭环。

## 5. 设计矛盾/冗余/缺失点
**可能的矛盾/重复**
- 商户入驻存在“旧版”与“新版”两套入口（/merchants/applications 与 /merchant/application），状态与权限模型不同，易造成管理与前端调用歧义。
- 骑手入驻也保留新旧两套入口（/rider/application 与 /rider/apply）。
- 角色/权限校验混用 Casbin、role 查询与中间件，可能导致权限规则不一致。

**潜在冗余**
- 部分列表接口直接返回 sqlc 结构体而非统一响应结构（例如会员列表），易造成字段泄露或结构不一致。

**可能缺失**
- 缺少“支付成功但任务入队失败”的自动补偿扫描任务（已有告警，但缺少自动恢复策略）。
- 对公共/私有上传路径的权限校验已实现，但对外链 URL 的兼容路径在各处处理不一致，建议统一规范。
