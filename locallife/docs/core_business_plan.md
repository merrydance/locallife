## 1. 背景与目标
- [x] 1.1 明确业务目标与成功标准
  - 目标说明（草案）：
    - 把复杂度集中到“堂食会话/扫码点餐”主线，形成统一的业务主流程。

  - 成功指标（草案）：

    - 订单类型与必填字段能区分堂食/外卖/预约（见 [api/order.go](api/order.go#L20-L120)）。
  - 已确认：将“多人扫码/AA/拼桌”纳入本阶段目标。
- [x] 1.2 界定范围与边界
  - 范围说明（草案）：
    - 堂食：扫码预检、开台、会话内下单与加菜。
    - 预约：预订支付模式、到店核销/转堂食。
    - 外卖：下单与支付入口、与堂食共享订单/支付状态。
  - 不做事项（草案）：
    - 不在本阶段重构配送或骑手调度细节。

## 5. 方案与策略（只做方向性选择）
- [x] 5.1 方案选型
  - 方案对比（草案）：
    - 方案 A：在现有用餐会话上引入“账单组”概念，统一多人扫码/AA/拼桌结算。
      - 优点：复用会话与订单结构，改动范围可控。
      - 风险：需要补齐结算规则与并发控制。
    - 方案 B：维持单订单模型，通过“订单合并规则”实现多人结算。
      - 优点：数据库改动小。
      - 风险：状态与日志复杂度高，难以统一。
  - 初步倾向：方案 A（统一账单组/会话模型）。
- [x] 5.2 决策结论
  - 结论（草案）：
    - 选择“会话 + 账单组”作为主模型，用以承载多人扫码/AA/拼桌。
    - 预约到店后必须合流到会话，并支持预点菜迁移与差量合并。

### T7 产出：测试用例清单（草案）

#### A. 并发与扫码
- 同一桌台同时扫码开台 → 仅创建一个会话。
- 多人同时加入账单组 → 成员不重复。

#### B. 结算与账单组
- 同伴扫码加入已有账单组 → 共同点单、单人买单。
- 选择“单独结算” → 创建新账单组互不影响。
- 账单组关闭后再扫码 → 按规则提示或新建账单组。

#### C. 预约与库存
- 预约支付成功 → 触发库存预留。
- 预约取消/过期 → 库存释放。
- 到店后改菜：新增校验库存，减少走退款。

#### D. 业务异常
- 同菜重复下单提示正确。
- 支付成功但未清台 → 会话仍保持开放。
- 厨房已开工 → 禁止减少菜品。

  ### T5 产出：预约到店后的差量合并/补单/退单规则（草案）

  #### A. 差量合并原则
  - 以“预约预点菜”为基线，形成到店后的“差量订单”。
  - 新增菜品 → 走加菜补单流程。
  - 减少菜品 → 走退单/退款流程。

  #### B. 关键规则
  - 已下发厨房/已制作的菜品不可减少或需走人工确认。
  - 变更前必须重新校验库存（新增部分）。
  - 变更后同步调整库存预留（释放或追加）。

  #### C. 结算口径
  - 追加菜品按当日售价计价。
  - 减少菜品按原价退回（或按商户规则）。
  - 若已支付：自动生成补单或退款单据。

  #### D. 操作边界
  - 允许变更的时间窗口：到店后至“厨房开工”前。
  - 超出窗口：只能追加，不允许减少。

  ### T4 产出：预约点菜库存预留/冻结规则（草案）

  #### A. 触发时机
  - 预约点菜在“支付成功”时触发库存预留/冻结。

  #### B. 规则口径
  - 以预约日期 + 时段为维度，预留对应菜品数量。
  - 若库存不足：支付前即校验并拒绝；支付后不可继续占用。
  - 取消/过期/未到店时释放预留库存。

  #### C. 与下单扣减关系
  - 预约预留是“冻结”，真正出餐或落单时再做实扣减。
  - 若到店后改菜：先释放原预留，再按差量重新预留。

  ### T3 产出：多人扫码加入会话/账单组流程与并发策略（草案）

  #### A. 加入流程（多人扫码）
  1. 用户扫码进入桌台 → 预检桌台/预约状态。
  2. 若会话存在：
    - 返回当前会话信息。
    - 默认加入当前账单组（同伴一人买单）。
    - 若用户选择“单独结算” → 创建新的账单组。
  3. 若会话不存在：
    - 发起开台请求（首次扫码者成为 owner）。
  4. 加菜与下单：
    - 每次下单绑定到当前账单组。
    - 下单前可拉取账单组已点菜品与支付状态用于提示。

  #### B. 并发策略（避免重复开台/重复加入）
  - 会话层并发：
    - 以 table_id 或 reservation_id 为互斥维度，保证只创建一个开放会话。
    - 若已存在会话，直接返回而不重复创建。
  - 账单组并发：
    - 以 dining_session_id 为维度，限制同一会话的账单组创建并发。
    - 用户加入账单组需幂等（同一 user_id 不重复插入）。

  #### C. 角色与权限（草案）
  - owner：可关闭账单组、发起整单支付。
  - member：可加菜/发起自己订单支付。

  #### D. 失败与冲突处理
  - 如果桌台已被他人开台且非本预约：提示冲突。
  - 如果账单组已关闭：创建新的账单组或提示不可加入（待业务决策）。
  - 默认规则：用户扫码时由前端提供“加入已有账单组 / 单独结算”选择。

  #### E. 会话结束与桌台占用规则（补充）
  - 支付不是会话终点：只要服务员未清台/未释放桌台，桌台仍保持占用。
  - 账单组可先关闭，但会话仍保持开放，直到清台/释放为止。

### T2 产出：账单组模型（草案）

#### A. 目的
- 支撑多人扫码/AA/拼桌的统一结算与状态回溯。
- 避免“多订单互相覆盖或难以合并”的复杂性。
- 支持多人同时加菜与共享点单信息（同菜提醒/同伴支付状态提示）。

#### B. 核心实体
1. BillingGroup（账单组）
   - 归属：dining_session_id
   - 字段建议：
     - id
     - dining_session_id
     - status（open/partially_paid/paid/closed）
     - total_amount / paid_amount / remaining_amount
     - created_at / updated_at / closed_at
2. BillingGroupMember（账单组成员）
   - 归属：billing_group_id
   - 字段建议：
     - user_id
     - join_at / leave_at
     - role（owner/member）
3. BillingGroupOrder（账单组订单关系）
   - 归属：billing_group_id
   - 字段建议：
     - order_id
     - amount
     - status（linked/paid/refunded）

#### C. 结算规则（草案）
- 整单支付：任何成员可一次性支付账单组剩余金额。
- AA 支付：按订单或按人均拆分，分笔支付累积到 paid_amount。
- 拼桌场景：同会话多个账单组可并存，但每个订单只能归属一个账单组。
- 同菜提示：账单组内已有相同菜品时，前端提示“同伴已点/已支付”。

#### D. 状态迁移（草案）
- open → partially_paid → paid → closed
- 若发生退款：paid → partially_paid 或 paid → closed（视业务规则）

#### E. 与现有模型的关系
- 账单组依附用餐会话（见 [api/dining_session.go](api/dining_session.go#L18-L326)）。
- 订单仍为履约与支付的基础实体（见 [api/order.go](api/order.go#L20-L120)）。

  ### T1 产出：堂食主流程 + 预约合流（文字版流程图）

  #### A. 堂食主流程（扫码点餐主线）
  1. 用户扫码进入桌台页面 → 发起桌台预检。
    - 若无预约占用 → 可直接开台。
    - 若有预约占用且非本人 → 提示不可用。
  2. 开台请求（桌台 ID，可选预约 ID）。
    - 若已有开放会话 → 直接返回会话。
    - 若无会话 → 创建用餐会话。
  3. 会话内下单/加菜。
    - 订单类型为堂食，绑定 table_id。
  4. 支付与结账。
    - 支付成功后进入履约流程（厨房/出餐/完成）。

  #### B. 预约合流流程（预约到店 → 堂食会话）
  1. 用户预约并支付（定金/全款）。
  2. 到店扫码后预检命中预约 → 返回预约信息。
  3. 开台时带 reservation_id：
    - 校验预约归属/时间窗口/状态（paid/confirmed/checked_in）。
    - 若已有预约订单且已支付 → 激活订单并进入履约。
    - 导入预约预点菜至会话（可选创建/复用购物车）。
  4. 到店后追加/改动：进入“差量合并/补单/退单”规则（T5 细化）。

  #### C. 关键证据位置（实现依据）
  - 桌台预检：见 [api/dining_session.go](api/dining_session.go#L86-L161)
  - 开台与预约校验：见 [api/dining_session.go](api/dining_session.go#L165-L326)
  - 预约状态与预点菜结构：见 [api/table_reservation.go](api/table_reservation.go#L20-L96)
  - 订单类型与桌台/预约字段：见 [api/order.go](api/order.go#L20-L120)
