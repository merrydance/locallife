# WebSocket 推送可靠性改造计划

目标：在不破坏现有推送能力的前提下，分阶段提升 WebSocket 推送的可靠性、可追溯性与扩展性。每一步完成后可勾选。

## 范围
- WebSocket 连接与心跳
- 消息协议与幂等
- 断线重连与回放
- 可靠投递与背压
- 监控与告警

## 阶段 0：基线确认
- [x] 确认当前 WS 仅用于通知推送，不承载位置上报
- [x] 盘点推送来源（新订单→商户；订单流→特定骑手群体；订单状态变更→用户/商户/骑手；平台告警→运营）
- [x] 明确“可靠送达”定义：有效恰好一次（服务端至少一次 + 客户端幂等去重）

## 阶段 1：消息协议与幂等
- [x] 为推送消息增加 message_id（UUID）与 sequence 字段
- [x] 客户端 ack 结构统一化（ack: {message_id, sequence, ts}）
- [x] 服务端维护短期 ack 记录（内存/Redis，TTL）
- [x] 基于 message_id 做幂等去重

## 阶段 2：断线重连与回放
- [x] 客户端重连策略（指数退避 + 抖动）
- [x] 支持携带 last_sequence 进行恢复
- [x] 服务端提供“消息回放”接口或 WS 协议支持
- [x] 回放仅限短期（例如 10–30 分钟）

## 阶段 3：可靠投递与背压
- [x] 发送队列落地（Redis Stream / List）
- [x] 消费端支持速率限制与丢弃策略
- [x] 连接侧背压策略：缓冲区告警 + 降级策略
- [x] 超时未 ack 的消息重试（有限次数）

## 阶段 4：可观测性与告警
- [x] 监控指标：在线数、发送成功/失败、丢弃数、重试数
- [x] 延迟指标：生产->发送->ack
- [x] 关键告警：推送失败率、队列堆积、连接抖动

## 阶段 5：回归与灰度
- [x] 单测：ack/重试/回放/幂等
- [x] 压测：高并发连接与消息突发
- [x] 灰度开关：按租户/区域/比例开启

## 说明
- 现阶段“实时提醒”已可用，但可靠性不足。
- 建议优先完成阶段 1–2，再评估是否需要阶段 3 的持久化投递。
