# 行为追溯裁决系统设计框架（讨论稿）

> 目标：替代信用分作为异常订单裁决核心；支持“先赔付、后治理、可追溯”。
> 状态：讨论用框架，待补充阈值与落地计划细节。

## 1. 目标与原则
- 正常订单零干预；只处理异常订单。
- 裁决以行为追溯为依据，要求可解释、可追溯。
- 体验优先：顾客索赔先赔付；裁决结果用于后续治理。
- 自动化唯一：取消任何人工干预，所有裁决由规则系统自动完成。
- 信用分相关计算/扣分逻辑与代码必须全部删除，不保留观测字段。

## 2. 范围与非目标
### 2.1 覆盖范围
- 索赔提交、食安、欺诈检测、申诉、恢复。
- 异常订单裁决与责任划分。

### 2.2 非目标
- 不在此版本继续强化“信用分扣分/积分”裁决逻辑。

## 3. 数据与信号
### 3.1 异常事件模型（最小字段集）
- 订单维度：order_id、订单类型、金额、运费、时间窗。
- 参与方：user_id、merchant_id、rider_id（可空）。
- 触发类型：claim_type / food_safety / delivery_exception / timeout / damage。
- 证据：图片/描述/地理/设备指纹/地址ID。
- 关联：历史异常、历史正常订单、相关参与方关系。

### 3.2 信号来源
- 纵向回溯：同一参与方的历史异常与正常交易。
- 横向对比：同一商户/骑手/区域的群体异常分布。
- 关联检测：设备/地址/时间窗/同伴行为聚集。

## 4. 行为追溯引擎
### 4.1 规则层（可解释）
- 时间集中：短时多次异常。
- 同一商户/骑手集中：异常集中度阈值。
- 高频异常：7天/30天窗口内异常次数。
- 比例异常：异常次数占总订单比例（如异常率阈值），先看7天窗口；若7天仅1单，则补算30天窗口作为补充分母。
- 关联模式：设备复用、地址聚类、协同异常。

### 4.2 对比层（基线对比）
- 异常占比 vs 正常订单基线（用户/商户/骑手维度）。
- 纵向趋势：异常频率是否显著上升。
- 横向偏差：相对同区域/同规模商户的异常偏差。

### 4.3 裁决策略
- 输出责任分配：商户/骑手/用户/不可归责。
- 输出建议动作：赔付来源、是否复核、是否进入观察名单。
- 需要保留裁决版本号，保证可回溯。

## 5. 裁决输出与动作
### 5.1 输出结构（建议）
- decision_id, decision_version, reason_codes
- responsible_party (merchant/rider/user/unknown)
- compensation_source (merchant/rider/platform)
- needs_review (true/false)
- evidence_required (true/false)
- trace_summary (可读说明)

### 5.2 动作策略
- 顾客索赔：先赔付（体验优先）。
- 商户/骑手：责任归属后执行扣款/冻结/通知/观察。
- 风控提示：命中关联模式时进入观察名单并记录。

## 6. 流程与时序
1) 异常事件提交
2) 规则层判定 + 行为回溯
3) 责任分配与赔付来源计算
4) 执行动作（赔付/冻结/通知）
5) 写入裁决记录与审计日志
6) 异步复核/申诉（必要时）

## 7. 申诉与复核机制
- 申诉与复核仅面向商户/骑手；不面向顾客。
- 不提供人工复核；申诉仅作为新证据输入源，进入规则再评估流程。
- 再评估结果仍由规则系统自动给出，并回写裁决记录。

## 8. 骑手接单资格（简化版方向）
- 从“积分体系”改为“事件阈值 + 观察窗口”。
- 规则示例：
  - N 天内超时/餐损达到阈值，暂停高值单资格。
  - 恢复条件：连续完成 M 单且无异常。
- 目标：规则少、可解释、易运营。

## 9. 可观测性与审计
- 指标：异常占比、裁决一致性、复核率、误判率。
- 审计：裁决理由、版本、证据链、执行动作可追溯。

## 9.1 配置项建议（可配置化）
- 时间窗口与次数阈值：7天/30天异常次数阈值。
- 比例阈值：异常率阈值（异常次数/总订单），支持“7天仅1单则补算30天”的口径。
- 设备/地址聚类阈值：用户数阈值、索赔数阈值。
- 协同索赔阈值：窗口时长、最少用户数。
- 处理策略：拒绝服务时长、冷却期、提醒频率。
- 申诉/复核：最大申诉次数、再评估冷却期。

### 9.2 默认值建议（可落地为配置项）
- 异常次数阈值：7天≥3；30天≥5。
- 异常率阈值：7天≥30%；30天≥20%。
- 7天仅1单：补算30天窗口（分母下限=30天总单数）。
- 拒绝外卖服务冷却期：14天。
- 恢复规则：冷却期内无新增异常且完成≥3单正常外卖。
- 设备复用：7天同设备≥3用户且索赔≥3。
- 地址聚类：7天同地址≥3用户且索赔≥3。
- 协同索赔：1小时内同商户≥3用户且有关联。
- 店铺后台提醒频率：同一顾客24小时内最多1次提示。
- 申诉再评估冷却：7天。
- `device_fingerprint`：哈希存储，90天留存。

### 9.3 配置落地方式
- 方案A：写入配置文件（适合早期快速迭代）。
- 方案B：平台配置表（支持灰度与按城市/门店分层配置）。

## 10. 落地与迁移建议
- 第一步：落地裁决记录与证据链，确保可回溯。
- 第二步：裁决完全切换为行为追溯规则输出。
- 第三步：删除信用分相关表/接口/任务与代码路径。

## 10.1 顾客侧策略（结论）
- 顾客索赔：无条件赔付（体验优先）。
- 达到阈值后：拒绝其外卖服务（拦截外卖下单），不涉及预订和堂食服务。
- 堂食提醒：当顾客达到拒绝服务阈值时，在店铺后台提示“该顾客有多次恶意索赔记录，谨慎服务”。
- 落地现状：当前“拒绝服务”基于信用分阈值已存在，但“堂食提醒”尚未落地，需要新增触发与展示链路。

## 12. 可执行落地方案（实体/字段/状态机/事件）
### 12.1 核心实体
- Decision（裁决记录）
- Evidence（证据）
- TraceSnapshot（行为追溯快照）
- Action（动作执行记录）
- Appeal（申诉）
- Blocklist（拒绝服务名单）

### 12.2 关键字段（最小集合）
- Decision: id, version, order_id, entity_ids(user/merchant/rider), reason_codes, responsible_party, compensation_source, decision_status, created_at
- Evidence: id, decision_id, type(image/text/location/device/address), payload, created_at
- TraceSnapshot: id, decision_id, window_days, abnormal_count, total_count, abnormal_rate, association_hits, created_at
- Action: id, decision_id, action_type(block/refund/notify/observe), target_entity, status, executed_at, detail
- Appeal: id, entity_type, entity_id, decision_id, reason, evidence, status, created_at, reeval_at
- Blocklist: id, entity_type, entity_id, reason_code, block_until, status

### 12.3 状态机（简化）
- Decision: pending -> decided -> executed -> archived
- Appeal: submitted -> reevaluating -> resolved
- Action: created -> running -> success/failed
- Blocklist: active -> expired/removed

### 12.4 事件流（事件驱动）
- claim.submitted
- fraud.pattern.detected
- decision.created
- action.executed
- appeal.submitted
- decision.reevaluated
- blocklist.updated

## 13. 防欺诈采集与使用的缺陷与改进
### 13.1 现有缺陷
- “设备指纹”在检测中使用，但实际采集仅见登录时的`device_id`，与指纹概念不一致。
- 缺少在下单/索赔时自动注入设备信息的链路（目前仅手动触发接口可传入）。
- 设备/地址信号未形成统一采集与回放的证据链条。

### 13.2 改进方案（纳入本次设计）
- 统一设备信号模型：保留`device_id`作为设备标识，并新增`device_fingerprint`用于更强指纹区分。
- 在下单与索赔入口自动采集并写入：设备标识、IP、User-Agent、地址ID。
- 形成证据链：采集信号与裁决记录绑定，进入`Evidence`与`TraceSnapshot`。
- 接入规则引擎：设备/地址聚类、协同索赔检测默认在索赔提交时自动触发，无需人工调用。

### 13.3 事务性原则
- 采集与落库应作为下单/索赔等主流程的事务内附属动作，保证证据与业务记录一致性。
- 通知、异步分析与重算可在事务外异步执行。
