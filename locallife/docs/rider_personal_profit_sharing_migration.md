# 骑手改造：个人零钱分账迁移方案（平台收付通）

## 目标
- 骑手侧从“二级商户开户 + 提现”迁移到“订单分账直达个人微信零钱”。
- 去除平台人工审核作为主流程依赖，保留异常处理旁路。

## 当前状态（已完成的最小修复）
- 已将自动审核通过后的骑手状态推进到 `pending_bindbank`，不再依赖平台人工审核接口推进开户状态。
- 位置：`db/sqlc/tx_rider_application.go`。

## 推荐目标架构
1. **准入层**：骑手实名 + 资质自动审核（已存在）。
2. **接收方层**：骑手绑定“微信支付个人接收方标识”（如 openid 维度）。
3. **分账层**：订单完成后按规则直接向骑手个人零钱分账。
4. **对账层**：分账成功/失败回调 + 重试补偿 + 审计。

## 分阶段实施

### Phase A：数据与接口准备
- 新增骑手分账接收方字段（建议）：
  - `riders.wechat_personal_receiver`（接收方账号）
  - `riders.receiver_status`（未绑定/已绑定/失效）
  - `riders.receiver_bound_at`
- 新增接口：
  - `POST /v1/rider/profit-sharing/receiver/bind`
  - `GET /v1/rider/profit-sharing/receiver/status`

### Phase B：支付分账链路切换
- 订单分账接收方从“骑手子商户”切换为“骑手个人零钱接收方”。
- 保留配置开关：
  - `RIDER_PROFIT_SHARING_MODE=sub_mch|personal_wallet`
- 逐区域灰度发布。

### Phase C：回调与补偿
- 分账失败分类处理：
  - 接收方未绑定/失效 → 通知骑手重新绑定
  - 风控拒绝/限额 → 自动重试 + 人工工单
- 增加幂等任务：
  - 失败重放队列 + 死信告警

### Phase D：下线旧开户链路
- 下线骑手 `applyment` 页面入口与主流程依赖。
- 保留旧接口一段时间用于兼容，后续再删除。

## 风险点清单
- 个人接收方可用性与风控拒绝率上升。
- 退款/分账回退在个人接收方场景下的处理差异。
- 高并发分账幂等与回调乱序。

## 验收标准
- 骑手从提交申请到可接单，无需平台人工审核动作。
- 骑手收益结算可在 T+0/T+1 规则内稳定到账零钱。
- 分账失败可追踪、可重试、可审计。
