# 修复清单（详细版）

> 目标：基于总体审查报告输出可执行的修复事项清单（包含已完成修复进度）。
> 分级：P0=必须修复；P1=高优先；P2=中优先；P3=低优先。

## P0（必须修复）

### P0-1 统一入驻链路（商户/骑手）
**问题**：商户与骑手存在“新版/旧版”两套入口，状态与权限模型不一致，易引发前端调用歧义与数据分裂。
- 现状示例：
  - 商户：/v1/merchants/applications 与 /v1/merchant/application
  - 骑手：/v1/rider/apply 与 /v1/rider/application

**修复目标**：确定唯一主流程 + 明确废弃路径与迁移策略。

**当前进度**：已完成旧接口移除与权限同步（代码与测试）。

**建议动作**：
1. 定义主流程与废弃流程的“最终去向”（保留/下线/转发）。
2. 直接删除旧接口逻辑与路由注册（不再保留兼容层），并同步文档说明。✅ 已完成（移除旧商户/骑手入驻路由与 handler，更新测试）。
3. 数据层统一到同一套表/状态机，不再并行维护两套写入路径。

**已完成**：
- Swagger 文档已移除旧入驻路径（商户/骑手旧版接口）。

**验收标准**：
- 同一用户的入驻状态在单一入口可读写、可追溯。
- 旧入口在文档与响应中清晰标注“deprecated”。

---

### P0-2 权限校验一致化
**问题**：角色查询、Casbin、中间件组合混用，存在一致性风险。

**修复目标**：建立统一的授权链路（单入口校验 + 统一上下文）。

**当前进度**：已将管理员商户审核接口改为统一 Casbin 入口，并将商户评价操作切换为商户中间件校验。

**建议动作**：
1. 统一封装 RBAC 校验中间件，禁止 handler 内部重复角色校验。
2. 将“商户/运营商/管理员”身份提取为统一上下文对象，避免多处重复查询。
3. 为所有敏感接口建立“权限矩阵清单”。

**已完成**：
- 管理员商户审核接口通过 `CasbinRoleMiddleware(RoleAdmin)` 统一校验。
- 商户评价相关接口通过 `MerchantStaffMiddleware("owner")` 统一校验，移除 handler 内角色查询。
- 商户优惠券/会员管理接口通过 `MerchantStaffMiddleware("owner", "manager")` 统一校验，移除 handler 内角色查询。

**已调整**：
- 移除管理员商户审核路由及实现（已由自动审核流程替代）。

**已完成**：
- 已手动同步 Swagger 文档移除管理员审核接口条目（因环境无法安装 `swag`）。

**说明**：
- `swag` 安装受网络限制失败，后续可在具备网络的环境重新生成以确保一致性。

**验收标准**：
- 任意敏感接口只经过统一授权入口，不在 handler 内做额外角色判断。

---

### P0-3 支付后续任务补偿
**问题**：支付回调已落库但任务入队失败时，仅告警，无自动补偿。

**修复目标**：引入补偿扫描与自动修复机制，确保订单一致性。

**当前进度**：已存在补偿调度（payment recovery 定时扫描并补偿入队）。

**建议动作**：
1. 增加定时任务扫描：已支付但未完成后续处理的订单。
2. 补偿任务具备幂等处理逻辑。
3. 若补偿失败，推送更高等级告警并记录可追踪工单标识。

**结论**：现有实现满足“定时扫描+入队补偿”基础要求，后续仅需核对告警等级与幂等覆盖面。

**验收标准**：
- 在任务队列不可用/短暂失败时，系统可自动恢复订单状态链路。

## P1（高优先）

### P1-1 统一响应结构
**问题**：部分接口直接返回 sqlc 结构体或不一致的响应格式。

**修复目标**：统一响应 envelope 与数据字段，避免接口间结构漂移。

**当前进度**：已启用统一响应中间件（`ResponseEnvelopeMiddleware`），覆盖 `/v1` JSON API。

**建议动作**：
1. 制定统一响应规范（code/message/data）。
2. 对列表、详情、分页结构统一字段命名（page_id/page_size/total）。
3. 禁止 handler 直接返回 db 模型。

**验收标准**：
- 全部 API 返回符合统一格式，文档与实际响应一致。

**已完成**：
- `/v1` 路由已开启 `{code,message,data}` 统一响应封装。

**结论**：本轮已完成 db model 直出清理与分页字段统一；剩余仅为非分页列表接口保持数组输出。

**本轮进度**：
- 代金券用户列表相关接口改为 DTO 响应，避免 sqlc row 直出（/v1/vouchers/me、/v1/vouchers/me/available、/v1/vouchers/available/{merchant_id}）。
- 订单统计接口改为 DTO 响应，移除 db.GetOrderStatsRow 直出与文档绑定。
- 会员列表与充值规则相关接口改为 DTO 响应，移除 sqlc 结构体直出。
- 商户财务订单与顾客统计列表增加统一分页字段（page_id/page_size/total/total_count），兼容旧字段。

### P1-3 上传路径规范化
**问题**：不同模块对上传 URL/相对路径处理不一致。

**修复目标**：统一存储路径与对外 URL 规范。

**建议动作**：
1. 统一 normalize 规则，禁止模块内自定义拼接。
2. 对外部 URL 进行统一白名单和签名策略处理。
3. 文档明确“公共/私有路径”规则。

**验收标准**：
- 所有上传/访问接口遵循统一规则，无重复分支逻辑。

**当前进度**：已对预定明细返回的菜品图片 URL 做规范化处理；补齐 Boss/员工/会员相关列表与详情的头像/Logo URL 规范化；补齐桌台二维码 URL 规范化。

**结论**：已完成上传路径规范化处理。

**修复目标**：集中管理状态迁移规则。

**当前进度**：已为配送相关动作（接单/取餐/配送/送达）与用户确认收货补齐状态机约束，并复用统一校验函数阻断非法跳转；预定确认/完成/取消/签到/未到店/起菜新增统一状态校验入口。

**验收标准**：

**修复目标**：补充结构化日志与指标标签。

**验收标准**：

### P2-3 通知一致性
**问题**：通知偏好/免打扰/推送状态分支逻辑分散。

**修复目标**：统一通知发送入口及幂等逻辑。

**当前进度**：API 层通知改为统一经 `SendNotification` 入口分发；通知任务处理侧补齐用户偏好与免打扰校验，推送结果继续落库标记。

**建议动作**：

**验收标准**：
