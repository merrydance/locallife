# Weapp 商户侧收付通进件真值对齐修复计划

日期：2026-04-28

## 1. 背景

本计划承接对 `weapp` 商户侧收付通进件页面组的 G3 级审查。当前主链路已经具备较好的方向：后端 `/v1/merchant/applyment/status` 会回查微信进件事实并返回签约链接、法人验证链接、汇款验证信息、二级商户号等状态字段；小程序侧也已经把进件页面状态集中到 `weapp/miniprogram/services/merchant-applyment-workflow.ts`。

但当前实现还没有完全达到目标状态：

1. 最终开通态没有展示后端结算账户真值，也没有修改结算账户入口。
2. 微信待办二维码页没有把“保存二维码、退出小程序、使用微信扫一扫从相册识别、按微信提示完成认证/签约”的操作路径清楚展示出来。
3. 提交进件后没有直接消费 `merchantBindBank` 返回的阶段性真值，而是丢弃响应后依赖首页刷新。
4. `merchant_confirmation` / `to_be_confirmed` 语义需要收敛到真实后端或微信官方状态，避免前端保留无可执行载体的伪阶段。

## 2. 目标状态

商户侧收付通进件应由后端真值驱动 UI，形成清晰状态机：

1. 未提交或可重提：展示资料提交入口和阻塞原因。
2. 提交中：只显示动作级 loading，不给页面级“处理中”停留态。
3. 微信待办阶段性终态：当后端返回 `sign_url`、`legal_validation_url` 或 `account_validation` 时，进入待办页；二维码类待办必须可保存，并说明退出小程序后用微信扫一扫从相册识别二维码继续处理。
4. 微信人工审核：这是必要业务中间态，可以作为页面状态展示；用户回到进件页时应刷新后端状态并看到微信审核状态。
5. 审核驳回或冻结：展示后端拒绝/冻结事实、可重提条件和下一步。
6. 最终开通：只有后端进件状态完成且商户支付能力激活后，才展示真正终态；终态页面必须以商户号和结算账户查询结果为核心，并提供修改结算账户入口。

## 3. 范围

### 3.1 本次纳入

- `weapp/miniprogram/api/merchant-applyment.ts`
- `weapp/miniprogram/api/merchant-settlement-account.ts`
- `weapp/miniprogram/services/merchant-applyment-workflow.ts`
- `weapp/miniprogram/pages/merchant/settings/applyment/index.*`
- `weapp/miniprogram/pages/merchant/settings/applyment/action/index.*`
- `weapp/miniprogram/pages/merchant/settings/applyment/submit/index.*`
- 必要时新增商户结算账户编辑页或复用已有银行表单领域组件。

### 3.2 暂不纳入

- 不重做后端微信进件提交流程。
- 不扩大支持主体范围。
- 不在小程序伪造后端没有返回的微信状态、链接或结算账户字段。
- 不把结算账户修改逻辑塞入通用银行表单组件；银行表单保持领域输入组件，页面或 workflow owner 负责请求和路由编排。

## 4. 架构原则

1. `merchant-applyment-workflow.ts` 继续作为进件状态到页面阶段/任务的 workflow owner。
2. 进件状态只消费 `getMerchantApplymentStatus()` 和 `merchantBindBank()` 返回的后端事实。
3. 结算账户最终态只消费 `getMerchantSettlementAccount()` 返回的后端事实。
4. Page 负责加载、渲染、动作转发、错误承接和路由；不要把微信状态语义散写在多个页面事件里。
5. QR 保存能力保留在 `weapp/miniprogram/utils/applyment-qrcode.ts`，待办页只负责展示和触发。
6. 非顾客侧页面保持 TDesign-first：状态、列表、按钮、空态、loading 优先用 TDesign 组件；不要新增顶部解释性大卡片或全宽 notice bar。

## 5. 分阶段修复计划

### 阶段 0：状态契约收口

目标：先确认并冻结小程序可识别的进件状态，不让 UI 继续承接伪状态。

改动点：

1. 梳理 `MerchantApplymentNormalizedStatus` 与后端状态映射。
2. 核对 `merchant_confirmation` / `to_be_confirmed` 是否有真实后端来源。
3. 若没有真实后端来源，将该分支降级为审核/等待刷新语义，或移除用户可操作任务表达。
4. 为每个状态写清楚：是否可提交、是否 action_required、是否可展示审核态、是否终态。

验收：

1. 前端状态枚举只表达后端或微信官方可证明的事实。
2. 不存在“页面提示用户去确认，但没有链接、二维码、汇款信息或后端动作”的阶段。
3. workflow 单测或最小断言覆盖签约、法人验证、汇款验证、审核中、完成、驳回。

Review checkpoint：确认状态机没有把普通处理中包装成页面级停留态；微信人工审核仍保留为必要展示态。

### 阶段 1：提交后直接进入阶段性终态

目标：提交进件后直接使用 `merchantBindBank` 响应决定下一屏，减少 Toast 后再刷新首页的绕路。

改动点：

1. 在 `submit/index.ts` 中保留提交中 loading 和防重入。
2. 接收 `merchantBindBank(requestPayload)` 返回值。
3. 用同一套 workflow builder 将提交响应转换为下一步阶段。
4. 若返回签约链接、法人验证链接或汇款验证信息，直接 redirect 到 action 页，并带上 preferred task。
5. 若返回审核中，回到进件首页展示微信审核状态。
6. 若返回最终完成，回到进件首页并触发最终态结算账户加载。
7. 提交失败只用一个反馈通道，并映射成中文业务提示。

验收：

1. 提交成功但需要签约/验证时，不再只显示“提交成功”然后等待首页二次刷新。
2. 重复点击提交不会发起重复请求。
3. 失败后表单保留用户输入，用户知道应该修改资料、稍后重试或联系平台。

Review checkpoint：确认没有前端伪成功；所有成功跳转都来自后端返回事实。

### 阶段 2：微信待办二维码页补齐操作闭环

目标：把阶段性终态做成清晰可执行的待办页。

改动点：

1. 在 `action/index.wxml` 渲染 `workflowView.currentTask.description` 与 `workflowView.currentTaskQRCodeHint`。
2. 二维码区明确区分签约二维码和法人验证二维码。
3. 底部动作区保留“保存二维码”和“刷新状态”。
4. 在二维码保存成功后，展示清晰中文提示：二维码已保存；请退出小程序，打开微信扫一扫，从相册选择该二维码并按微信提示完成。
5. 若平台支持且产品确认，可以增加“退出小程序”按钮；否则只做明确操作指引，不强行调用平台 API。
6. `bank_transfer_validation` 保持结构化汇款信息，不渲染二维码保存按钮。
7. `onShow` 回到待办页时继续刷新状态，扫码完成后能进入审核中或最终态。

验收：

1. 用户能在待办页看懂当前是签约、法人验证还是汇款验证。
2. 二维码待办用户知道为什么要保存、保存后去哪里、回来后页面会如何变化。
3. 待办页不使用全局 notice bar，不把说明做成顶部大卡片。

Review checkpoint：确认阶段性终态不是 loading，也不是笼统“处理中”；它必须提供真实下一步。

### 阶段 3：最终开通态接入结算账户真值

目标：最终态不再只是“已开通 + 子商户号”，而是展示真实可用的资金账户信息。

改动点：

1. 在进件首页进入 `opened` 阶段时加载 `getMerchantSettlementAccount()`。
2. 加入独立的 settlement account view model，转换后端字段：账户状态、账户类型、开户银行、支行、掩码账号、校验状态、失败原因、最新修改申请号。
3. 首屏加载策略：先加载进件状态；只有进件为最终开通时再加载结算账户，避免无意义请求。
4. 结算账户加载中只影响最终态局部区域，不把整页打回白屏。
5. 结算账户加载失败时保留“商户号已开通”的可信事实，并在结算账户区显示可重试错误态。
6. 页面展示商户号、结算账户状态和结算账户详情。
7. 根据 `getSettlementAccountStatusView()` 的 `canEditSettlementAccount` 显示“修改结算账户”入口。

验收：

1. 最终态主信息包含二级商户号和结算账户查询结果。
2. 结算账户接口失败不会误导用户为未开通或未提交。
3. 修改入口只在后端账户状态允许时展示。
4. 没有从进件状态本地拼装结算账户信息。

Review checkpoint：确认最终态展示的是后端结算账户真值，不是表单草稿或旧提交参数。

### 阶段 4：修改结算账户入口与申请结果承接

目标：让最终态的“修改结算账户”形成完整可用链路。

改动点：

1. 确认是否已有商户结算账户编辑页；若没有，新增 `pages/merchant/settings/applyment/settlement-account/edit` 或更合适的资金账户路径。
2. 复用 `applyment-bank-form` 的银行选择、支行选择和账户字段输入能力，但由新页面负责 `modifyMerchantSettlementAccount()` 请求。
3. 修改提交后使用 `application_no` 进入申请状态承接，而不是 Toast-only。
4. 若后端返回修改申请审核中，展示“微信正在审核结算账户修改”的必要中间态。
5. 修改申请查询使用 `getMerchantSettlementApplication(applicationNo)`，并能从最终态入口查看最近申请状态。
6. 修改成功后回到最终态并刷新 `getMerchantSettlementAccount()`。

验收：

1. 修改结算账户入口可达、可提交、可回查。
2. 修改申请审核中是明确的微信处理状态，不误报为修改成功。
3. 修改失败展示失败原因和下一步，不暴露微信原始英文或内部字段。

Review checkpoint：确认修改链路没有把“申请已提交”当成“账户已修改完成”。

### 阶段 5：重入、弱网和错误承接收口

目标：让用户从微信扫码、微信审核、页面重进、接口失败中都能回到可信状态。

改动点：

1. 统一首页、待办页、提交页的刷新标记和 `onShow` 回读策略。
2. action_required 页面每次 `onShow` 刷新仍保留，但避免重型无关请求。
3. opened 页面只在需要时加载结算账户，不在非终态预热。
4. 首屏失败显示页内 `t-empty` + 重试。
5. 局部刷新失败保留上次可信数据，并显示局部错误。
6. 统一用户可见错误文案，避免透出第三方或后端原始诊断。
7. 明确未知结果：提交超时、刷新失败、微信状态暂未同步时，用户看到的是“状态同步失败/稍后刷新”，不是成功或失败。

验收：

1. 从微信扫码完成后返回小程序，再进入收付通进件，页面展示微信审核状态或最终态。
2. 弱网刷新失败不会抹掉已知商户号或已知账户信息。
3. 重复保存二维码、重复刷新、重复提交都不会造成多次高风险动作。

Review checkpoint：确认没有 Toast-only 首屏失败，没有默认值伪装真实结果。

### 阶段 6：门禁、复审和提交

目标：按 G3 变更完成质量闭环。

执行命令：

1. 从 `weapp/` 执行 `npm run quality:check`。
2. 执行 `git diff --check -- weapp`。
3. 如新增页面或 app.json 路由，确认小程序构建/门禁覆盖页面注册。
4. 如改动后端 contract 或路由，再执行对应 backend 生成和测试；若只改 weapp，不跑 backend 生成。

人工验证矩阵：

1. 未提交状态：可进入提交页。
2. 提交后返回签约链接：直接进入签约二维码待办页。
3. 提交后返回法人验证链接：直接进入法人验证二维码待办页。
4. 提交后返回汇款验证信息：进入汇款验证待办页。
5. 扫码后返回：进入审核中状态。
6. 审核通过：展示商户号和结算账户信息。
7. 结算账户查询失败：保留开通事实并允许局部重试。
8. 修改结算账户：提交后展示修改申请状态，完成后刷新最终态。
9. 驳回：展示驳回原因和重新提交入口。

交付说明必须包含：

1. 风险等级：G3。
2. 自动化命令结果。
3. 人工验证路径。
4. 未验证路径及剩余风险。
5. 是否涉及后端 contract 缺口。

## 6. 推荐落地顺序

建议按以下节奏推进，每一阶段完成后先 review 再进入下一阶段：

1. 阶段 0 + 阶段 1：先收口状态和提交后跳转。
2. 阶段 2：补齐待办二维码交互闭环。
3. 阶段 3：接入最终态结算账户真值。
4. 阶段 4：补齐修改结算账户链路。
5. 阶段 5 + 阶段 6：恢复路径、门禁和最终复审。

如果需要进一步压缩首个可交付版本，可以先交付阶段 0-3，确保进件主链路和最终态正确；阶段 4 作为第二个 PR/提交继续补齐修改入口的完整编辑与回查。

## 7. 风险与待确认项

1. `merchant_confirmation` / `to_be_confirmed` 是否仍有后端真实来源：需要在阶段 0 明确。若没有，应移除可操作表达。
2. 小程序是否允许或是否产品上需要调用退出小程序能力：阶段 2 只把它作为可选项，不作为强依赖。
3. 结算账户修改页路径归属：可以放在收付通进件页面组下，也可以放入商户资金账户页面组；需要以用户任务连续性为准。
4. 后端结算账户接口返回的是微信查询事实，页面不得用进件表单草稿替代。
5. 若微信状态同步延迟，页面只能展示“同步中/审核中/刷新失败”等可信状态，不能提前展示开通成功。

## 8. 完成定义

本计划完成时，应满足：

1. 商户侧收付通进件所有页面状态由后端事实驱动。
2. 普通处理中不作为页面级终态停留；微信人工审核作为必要中间态展示。
3. 签约/法人验证二维码是阶段性终态，支持保存并给出明确退出小程序和微信扫一扫路径。
4. 最终终态展示商户号、结算账户信息和修改结算账户入口。
5. 修改结算账户不是假成功，必须展示申请状态并回查最终事实。
6. `npm run quality:check` 通过，且复审无阻断问题。
