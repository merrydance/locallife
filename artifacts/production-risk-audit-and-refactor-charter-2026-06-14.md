# 生产风险审计与重构任务宪章

日期：2026-06-14

## 背景

当前系统已经达到不能依赖一次性全仓库阅读来判断风险的规模。直接把整个系统塞进单次上下文，会导致判断漂移、遗漏链路、误把局部代码当成全局事实，也会让重构方案变得过大而不可验证。

本任务的出发点是：生产级系统中真正容易造成事故的问题，集中出现在跨时间、跨请求、跨权限边界的状态变化上。已识别的核心风险维度包括：

- 状态时序
- 幂等重试
- 权限边界
- 事务一致性
- 外部依赖
- 发布配置

这些维度不一次性并行展开。每一阶段只聚焦一个风险维度，先完成该维度下的高风险业务流审计、修正、验证和台账沉淀，再进入下一个维度。

## 文档定位与优先级

本文档是 `artifacts/` 下的长期任务锚点，用于保留本次生产风险审计与重构工作的背景、方法论、阶段边界和进度。它不替代仓库入口规则、目录级 `AGENTS.md`、`.github/instructions/`、`.github/prompts/` 或 `.github/standards/`。

任何新会话、压缩后会话、交接会话或分支会话，都必须先按仓库规则完成路由：

1. 读取仓库根目录 `AGENTS.md`。
2. 读取 `.github/copilot-instructions.md` 和 `.github/README.md`。
3. 根据目标路径打开最小匹配的 `.github/instructions/*.instructions.md`。
4. 当任务进入实现、bugfix、review、integration-test、payment-domain、SQL review 或 takeover 时，再打开最小匹配的 `.github/prompts/*.prompt.md`。
5. 当业务流触及支付、退款、分账、提现、回调、worker、scheduler、authn/authz、tenant 边界等高风险路径时，再打开对应 domain README 或 backend 标准。
6. 完成以上路由后，再读取本文档、当前阶段台账和当前业务流材料。

## 总目标

建立一套可持续的生产风险审计与重构工作流，逐步把系统中高风险业务流的状态变化、幂等保护、权限边界、事务一致性、外部依赖处理和发布配置风险梳理清楚，并在有证据支持的地方进行结构性修正或重构。

最终目标不是“打补丁式修几个点”，也不是“大爆炸重写系统”，而是：

- 用业务流作为最小审计与交付单位。
- 用风险维度作为阶段主题，避免多个问题域混在一起。
- 用 codegraph、人工业务链和代码验证相互校准。
- 用审计台账承接上下文，避免依赖单次会话记忆。
- 用测试、约束、迁移图和运行路径证据支撑每次修正。
- 在同类问题累计后再抽象公共机制，避免过早重构。

## 核心原则

### 1. 以重构视角审计，以证据驱动重构

审计时要站在系统重构的高度看问题，例如是否需要统一状态机、集中状态迁移函数、状态转移表、事件日志、条件更新、领域服务边界或补偿任务约束。

但重构必须由具体证据触发。不能因为某个主题重要，就直接宣布全系统重写。每次重构都要能回答：

- 当前真实生产路径是什么？
- 已确认的状态时序问题在哪里？
- 局部修复为什么不够，或为什么已经足够？
- 重构后的边界、入口、约束和测试是什么？
- 如何证明没有引入新的状态跳转、越权或重复执行路径？

### 2. 主题分阶段，业务流分批交付

风险主题按阶段推进：

1. 状态时序
2. 幂等重试
3. 权限边界
4. 事务一致性
5. 外部依赖
6. 发布配置

每个主题内部不做全系统横扫，而是按业务流推进。例如“状态时序”阶段可以拆成：

- 支付成功回调到订单状态更新
- 退款申请到退款成功或失败
- 商家接单、拒单、取消、完成
- 配送或履约状态推进
- 预约状态推进
- worker 或 scheduler 补偿任务的状态推进

每个业务流都必须形成闭环：定位入口、确认调用链、画出状态迁移、标记风险、决定修正或重构、补验证、更新台账。

### 3. Codegraph 是索引，不是事实终点

codegraph 产出非常有价值，用于快速缩小入口、函数链、调用边界、SQL、worker 和外部依赖范围。

但 codegraph 只能作为索引和假设地图。最终判断仍要回到代码、SQL、事务边界、运行时注入、队列调度、回调重试、测试和业务规则中确认。

人工穷尽的用户角色业务链也同样重要，但它也是待验证假设。审计时要同时追问：

- 业务上这个角色是否应该能执行该动作？
- 代码上是否真的限制了这个角色和资源归属？
- 是否存在 API、worker、callback、scheduler 等绕过主路径的入口？

### 4. 不靠上下文记忆，靠持久化台账续接

每个业务流审计后都要沉淀到台账。后续会话优先读取台账和当前业务流相关代码，而不是重新加载全系统。

每条业务流台账至少记录：

- 业务流名称
- 风险主题
- 角色和资源归属
- 入口列表，包括 API、callback、worker、scheduler
- codegraph 子图或关键调用链
- 关键状态和允许迁移
- 关键表、SQL、唯一约束和条件更新
- 已确认的状态时序风险
- 暂存但不展开的其他风险，例如幂等、权限、事务
- 修正或重构决策
- 已运行验证
- 未验证项和残余风险
- 下一步建议

默认台账位置：

- 阶段总账：`artifacts/production-risk-audit/<risk-theme>-ledger-YYYY-MM.md`
- 单流材料：`artifacts/production-risk-audit/flows/<risk-theme>-<flow-slug>-YYYY-MM-DD.md`
- codegraph 子图或边文件：优先复用 `artifacts/codegraph/` 下已审阅材料；如需新增，按该目录既有约定保存。

第一阶段状态时序默认使用：

- `artifacts/production-risk-audit/state-sequencing-ledger-2026-06.md`
- `artifacts/production-risk-audit/flows/state-sequencing-<flow-slug>-2026-06-14.md`

### 5. 审计和修复分阶段，默认先只读

每个业务流默认分成四个关口：

1. 审计：只读生产代码，允许新增或更新审计文档、台账和证据索引。
2. 修复或重构计划：基于审计证据说明范围、方案、触达文件、验证命令、回滚或残余风险。
3. 实施：用户明确批准计划后，才修改生产代码、SQL、迁移、生成物或测试。
4. 关闭：运行最小但充分的验证，更新台账，记录未覆盖项和下一步。

除非用户在同一轮明确要求直接修复，否则审计阶段不得修改生产代码。即使用户要求直接修复，也必须先给出最小修复意图、变更范围和验证方式，再动手改代码。

不允许在一个业务流审计或修复中顺手修改相邻领域。发现相邻问题时，只登记到当前台账的“旁路风险”或对应风险维度 backlog，等进入相应主题或业务流时再处理。

## 第一阶段：状态时序

第一阶段只聚焦状态时序。重点问题包括：

- 是否存在非法状态跳转。
- 是否存在重复写状态或多个入口写同一状态。
- 是否存在 API、callback、worker、scheduler 之间的乱序覆盖。
- 状态更新是否限制旧状态。
- 状态推进是否和副作用混在一个不可恢复的步骤里。
- 失败、超时、未知结果是否会留下半完成状态。
- 补偿任务是否可能推进过期状态。
- 测试是否覆盖重复、乱序、并发、过期事件和非法迁移。

幂等、权限、事务、外部依赖问题如果在状态时序审计中被发现，先记录到台账，除非它们阻塞当前状态时序判断，否则不展开修复，避免阶段范围失控。

“阻塞当前状态时序判断”只包括以下情况：

- 不确认该旁路问题，就无法判断当前状态迁移是否真实发生、是否允许发生、是否可能乱序覆盖。
- 不确认该旁路问题，就无法定位状态写入的真实入口或事务边界。
- 不确认该旁路问题，就无法设计当前状态时序的最小验证。

如果旁路问题只是会影响后续幂等、权限、事务、外部依赖或发布配置治理，但不影响当前状态迁移判断，则只登记，不展开。

第一阶段完成标准：

- 已列出状态时序阶段的高风险业务流清单，并标记优先级。
- 每条纳入本阶段的业务流都有审计结论，结论至少包含入口、状态迁移、状态写入点、风险判断和验证状态。
- P0/P1 状态时序风险已修复并验证，或已记录明确 owner、阻塞原因、临时缓解和后续计划。
- 旁路发现已归入对应风险维度 backlog，没有混入当前阶段无边界处理。
- 阶段总账已更新，能支持新会话从台账和本文档恢复上下文。

## 推荐试点业务流

推荐从“支付成功回调 -> 订单状态更新 -> 后续履约或分账触发”开始。

选择理由：

- 支付回调天然存在重复、延迟和乱序。
- 订单状态更新属于高价值状态迁移。
- 后续履约、分账、通知等副作用容易被状态时序问题放大。
- 该流可以同时暴露幂等、事务和外部依赖风险，但第一阶段只把它们作为旁路发现记录。

第一轮试点停止点：

- 进入范围：支付成功回调如何验证并推进订单状态；订单状态推进是否受旧状态限制；状态推进后是否可靠产生后续触发信号或记录。
- 不进入范围：履约内部状态机、分账内部状态机、通知内部重试策略、支付 provider 全量合同校验。
- 如果后续触发信号缺失或重复本身会影响订单状态是否已经安全推进，则记录为状态时序风险；如果只影响后续领域自身处理，则登记为旁路风险。

试点输入可以是：

- 支付成功回调的 codegraph 子图。
- 人工整理的支付成功业务链。
- 相关入口、函数、SQL、worker、测试文件清单。

## 单个业务流审计模板

每个业务流按以下顺序执行：

1. 明确业务流名称、角色、资源和目标状态变化。
2. 读取 codegraph 子图或业务链，形成候选路径。
3. 回到代码确认真实入口，包括 API、callback、worker、scheduler。
4. 找到状态常量、状态写入点、SQL 条件、事务边界和后续副作用。
5. 画出现有状态迁移表或迁移图。
6. 检查非法跳转、乱序覆盖、重复推进、过期事件和多入口竞争。
7. 标记旁路风险，但不把其他主题扩展开。
8. 决定是局部修正还是结构性重构。
9. 补最小但有效的验证，包括状态迁移测试、重复/乱序事件测试或集成测试。
10. 更新台账，记录已验证结论和残余风险。

## 重构决策标准

优先考虑结构性重构的信号：

- 同一业务实体的状态由多个模块直接写入。
- 状态迁移规则散落在 handler、logic、worker、SQL 中。
- 状态更新没有统一的旧状态条件。
- 关键副作用和状态推进没有清晰的事务或恢复边界。
- 测试难以表达“允许迁移”和“禁止迁移”。
- 新增业务状态时需要同时改很多互不感知的地方。

可以先局部修复的信号：

- 问题集中在单一入口或单一 SQL 条件缺失。
- 状态规则本身清晰，只是某处没有复用已有保护。
- 能通过小范围测试证明风险已收敛。
- 公共抽象会明显超过当前业务流需要。

## 上下文续接规则

后续任何会话接续本任务时，先按仓库入口完成路由，再读取本文档、当前阶段台账和当前业务流输入材料。

续接时不要默认已经记住历史上下文。应重新确认：

- 当前风险阶段是什么。
- 当前业务流是什么。
- 上一轮已经验证了什么。
- 哪些问题只是记录，尚未进入修复范围。
- 是否已经决定局部修复或结构性重构。
- 本轮最小验证命令是什么。

## 暂不做的事

- 不一次性审完整个系统。
- 不同时展开六类风险。
- 不在没有证据的情况下重写公共架构。
- 不把 codegraph 或大模型生成的业务链直接当成最终事实。
- 不把审计发现只留在对话上下文中。
- 不顺手改无关领域或相邻模块。
- 不在审计阶段默认修改生产代码。
- 不把“支付成功回调”试点扩大到履约、分账或通知的内部状态机。
