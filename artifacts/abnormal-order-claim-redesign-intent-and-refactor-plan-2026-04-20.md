# 异常订单顾客索赔重构设计意图与改造取舍

## 1. 目标

本文不再重复当前代码审计细节，而是把已经确认的业务意图收敛成一份可执行的重构判断标准。

要回答的问题只有三个：

1. 我们现在真正想要的异常订单顾客索赔链是什么。
2. 现有实现里哪些东西应该保留。
3. 现有实现里哪些东西应该删除、降级或重写。

## 2. 设计意图

## 2.1 这条链的本质

异常订单顾客索赔，不是“顾客一提交，平台就自动赔”。

它应该是一条两阶段链：

1. 判责阶段
2. 赔偿阶段

只有判责完成，才进入理赔执行。

## 2.2 判责阶段的核心原则

判责阶段应以当前索赔订单为锚点，而不是以顾客历史索赔集合为锚点。

判责要同时回溯三方：

1. 当前顾客
2. 当前商户
3. 当前骑手

并且必须使用同一个观察窗口来比较三方在同类问题上的行为表现。

## 2.3 `lookback` 的目标角色

`lookback` 不应再是“顾客历史索赔相关性分析器”，而应成为正式判责层的核心输入之一。

它要回答的不是：

1. 顾客是否总是对同一商户索赔
2. 顾客是否总是对同一骑手索赔

而应回答：

1. 顾客在同一时间窗口内的相似异常索赔历史是什么
2. 当前商户在同一时间窗口内被其他顾客索赔的历史是什么
3. 当前骑手在同一时间窗口内被其他顾客索赔的历史是什么

## 2.4 赔偿阶段的核心原则

赔偿阶段不负责再做判责。

赔偿阶段只消费判责结果，并做三类动作：

1. 平台赔付给顾客
2. 向商户或骑手生成追偿单
3. 在顾客高风险仍坚持索赔时执行限制名单与停服

## 2.5 对顾客的产品语义

顾客不应该在提交后立即看到“赔付处理中”作为主语义。

正确语义应是：

1. 平台已完成判责
2. 若支持赔偿，请等待赔偿到账
3. 若判责认为顾客风险高，则先展示原因，再让顾客决定是否继续索赔

## 3. 现有实现里应该保留的东西

## 3.1 行为决策持久化对象保留

以下对象方向是对的，应保留：

1. `behavior_decision`
2. `behavior_trace_snapshots`
3. `behavior_decision_effects`
4. `behavior_actions`
5. `claim_recovery`

理由：

这些对象已经把“判责事实”和“赔偿动作”区分成了正式可审计对象，具备长期演进基础。

## 3.2 赔付 outbox 机制保留

当前赔付不是 API 同步直接转账，而是先写 payout action，再由 scheduler 和支付回调推进终态。

这个方向应保留。

理由：

1. 真实支付动作天然适合异步执行。
2. 可以做重试、幂等和回调收口。
3. 这正符合“赔偿阶段消费判责结果”的目标。

## 3.3 追偿单对象保留

向商户或骑手生成追偿单的能力应保留。

但它应该只在判责阶段输出“责任方 + 追偿依据”之后才创建，而不是和判责同事务强耦合提前落下所有动作。

## 3.4 行为限制名单与停服能力保留

当前 `BehaviorBlocklist` 及其对外卖服务的拦截能力应保留。

理由：

1. 它已经提供了“赔付后停服”的技术底座。
2. 这部分不是方向错误，而是触发时机和触发条件需要调整。

## 3.5 设备/地址关联能力保留

共享设备、共享地址等关联关系，本质上仍然是判责的重要辅助证据。

这部分应保留，但它应从“用户评分信号”转成“正式判责证据”中的一类输入。

## 4. 现有实现里应该删除或降级的东西

## 4.1 删除“顾客历史索赔集合主导”的旧 lookback 设计

现有 [locallife/algorithm/lookback_checker.go](locallife/algorithm/lookback_checker.go) 的主轴不应保留。

应删除或彻底重写的点：

1. 先按顾客历史索赔拉出所有历史订单
2. 再从历史订单里归纳所有商户和骑手
3. 再判断顾客是否集中针对同一商户或同一骑手

理由：

这不是当前订单中心的三方回溯模型，会把判责问题误导成“顾客是否在针对某个对象”。

## 4.2 降级 API 前置轻量用户分层逻辑

[locallife/algorithm/claim_auto_approval.go](locallife/algorithm/claim_auto_approval.go) 里基于 `GetUserBehaviorStats` 的这套四档分流，不应继续承担正式判责职责。

应降级为：

1. 轻量提示
2. 资格校验
3. 判责任务启动前的辅助预警

不应继续在这里决定：

1. 是否平台兜底
2. 是否用户限制服务
3. 是否形成正式责任结论

## 4.3 删除“评分桥接主导最终决策”的定位

事务层当前最核心的问题，不是“算了分”，而是“让分数成为最终责任修正的主导”。

应该删除的是这个定位，不一定是所有分值字段本身。

具体要删的是：

1. 低责任分就平台兜底的主路径逻辑
2. 低置信度就平台兜底的主路径逻辑
3. 用户风险分阈值直接触发正式限制的主路径逻辑

理由：

正式责任应由三方回溯事实主导，分值最多做辅助排序或置信提示。

## 4.4 删除“提交即赔偿处理中”的外部语义

当前接口返回 `auto-adjudicated + processing` 的语义应删除。

理由：

这会把判责阶段和赔偿阶段压扁，导致外部产品语义错误。

## 5. 现有实现里应该重写的东西

## 5.1 重写 `lookback`

新的 `lookback` 应改成“当前订单中心三方窗口回溯”。

输入锚点：

1. 当前索赔订单 ID
2. 当前顾客 ID
3. 当前商户 ID
4. 当前骑手 ID
5. 当前异常类型

输出结构至少包含：

1. 顾客窗口摘要
2. 商户窗口摘要
3. 骑手窗口摘要
4. 三方对照结论
5. 面向用户的解释文本
6. 面向平台的判责原因码

## 5.2 重写正式判责层

正式判责层应只有一层，不应再出现“API 前先判一轮，事务里再用另一套规则修一轮”的双判责结构。

推荐做法：

1. 第一层只准备上下文和证据输入。
2. 正式判责器统一消费：
   - lookback 三方回溯结果
   - effect summary
   - 设备/地址关联
   - 履约责任事实
3. 统一产出：
   - 判责结果
   - 赔偿决策
   - 追偿决策
   - 用户通知文案
   - 是否限制服务

## 5.3 重写“责任不清晰”的默认归宿

当前“不清晰 -> 平台兜底”的逻辑应重写成：

1. 商品质量类默认商户兜责
2. 代取履约类默认骑手兜责

平台不再作为责任不清晰时的主默认承接方。

## 5.4 重写顾客高风险路径

当前系统会在主链里直接把高风险顾客推进到“平台赔付 + 用户限制”。

目标路径应改成：

1. 先向顾客展示判责原因
2. 顾客选择撤回，则关闭
3. 顾客确认继续，则执行赔付
4. 赔付完成后纳入限制名单并停服

也就是把“确认继续索赔”补成显式业务节点。

## 5.5 重写外部状态机

当前外部状态表达没有把“判责已完成”和“赔偿处理中”分开。

应改成：

1. `pending_platform_review`
2. `adjudicated`
3. `awaiting_compensation`
4. `compensating`
5. `compensated`
6. `warned_waiting_customer_confirmation`
7. `restricted_compensated`
8. `closed`

## 6. 具体保留/修改/删除清单

## 6.1 应保留

1. `behavior_decision` 及其快照结构
2. `behavior_actions` 作为 payout/recovery/block/notify 的 outbox
3. `claim_recovery` 追偿单能力
4. `BehaviorBlocklist` 与下单停服能力
5. 设备/地址关联查询能力
6. 支付回调与赔付 scheduler

## 6.2 应删除或退出主链

1. 现有顾客中心 `lookback` 逻辑
2. `CheckUserClaimBehavior` 承担正式判责职责的定位
3. “低责任分/低置信度 => 平台兜底”作为默认主规则
4. 提交索赔即返回赔付处理中作为外部主语义

## 6.3 应重写

1. `lookback` 的输入/输出模型
2. 正式判责器的输入结构
3. 高风险顾客确认继续索赔路径
4. 判责结果通知用户的产品链路
5. 判责与赔偿的状态边界

## 7. 建议的落地顺序

## 第一步：先改判责模型，不动赔付执行底座

先重写：

1. `lookback`
2. 正式判责器
3. 判责输出结构

暂时不动：

1. payout scheduler
2. transfer callback
3. recovery payment 执行链

## 第二步：拆开判责阶段与赔偿阶段

把当前“提交即建 payout/recovery/block/notify action”改成：

1. 提交 -> 只建 claim + adjudication job/context
2. 判责完成 -> 对用户发布判责结果
3. 需要赔偿时 -> 再建 payout/recovery action

## 第三步：接入顾客确认继续索赔节点

只在用户高风险路径下增加显式确认节点。

## 第四步：最后清理旧规则

等新判责链稳定后，再清掉：

1. 旧 `CheckUserClaimBehavior` 主判责逻辑
2. 旧 `promoteBehaviorPlatformFallback` 主默认兜底逻辑
3. 旧顾客中心相关性 lookback

## 8. 一句话结论

这次重构不是把整个异常索赔系统推倒重来，而是：

保留正式持久化、赔付 outbox、追偿单、限制名单这些底座；删除顾客中心旧 lookback 和“双判责 + 平台默认兜底”这些错误主轴；重写成“当前订单中心三方回溯 + 单一正式判责层 + 判责与赔偿分阶段”的目标链路。
