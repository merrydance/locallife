## Coverage Boundary

Current merchant-side Mini Program plus Flutter App state-changing workflow inventory is 17 flows. Every flow has a per-flow slice, an edges JSON file, and a `Branch Exhaustion` section that records entry branches, request branches, backend state branches, async paths, failure/retry paths, readers, authorization, zombie/unreachable candidates, and test-proof gaps.

The following merchant surfaces were intentionally classified as non-flow or already-owned entry points instead of adding an 18th flow:

- `weapp/miniprogram/pages/merchant/dashboard/**` and `weapp/miniprogram/pages/merchant/config/**` are entry aggregators. Their actions are assigned to the concrete flows below, such as manual open status, App binding, marketing rules, membership settings, device/display config, and business hours.
- `weapp/miniprogram/pages/merchant/stats/**` is read-only analytics in the current traced Mini Program scope. It does not own merchant-visible state mutation, so it is outside the state-changing workflow inventory unless a future write action is added.
- Flutter App flow coverage is limited to the features that exist under `merchant_app/lib/features/**`: bind/auth/device registration, working status, order actions/alerts, local notification settings, BLE printing, and table management. No App finance, onboarding, staff/group, marketing, dish, combo, membership, review, or profile-edit write entry was found in the current trace.
- Web merchant console is explicitly out of this audit round by user scope; any Web-only workflow should be handled in a separate pass.

This coverage statement does not mean the product has no defects. It means the currently discovered Mini Program/App merchant-side state-changing workflows are mapped to durable truth paths and the remaining defects, zombie candidates, idempotency gaps, authz decisions, and missing tests are registered below and in the per-flow slices.

## Scope

This audit covers merchant-side Mini Program state-changing flows and their backend truth paths. Each row is a product workflow, not just a page or API inventory.

Primary frontend scope:

- `weapp/miniprogram/pages/merchant/**`
- `weapp/miniprogram/pages/merchant/_main_shared/api/**`
- `weapp/miniprogram/pages/merchant/_services/**`
- shared API wrappers used by merchant pages, especially `weapp/miniprogram/api/**`
- `merchant_app/lib/features/**` where the Flutter merchant App reads or mutates the same merchant-side truth

Primary backend scope:

- `locallife/api/server.go`
- `locallife/api/**`
- `locallife/logic/**`
- `locallife/db/query/**`
- `locallife/db/sqlc/tx_*.go`
- `locallife/scheduler/**`
- `locallife/worker/**`
- websocket/outbox/callback paths when they affect merchant-visible state
