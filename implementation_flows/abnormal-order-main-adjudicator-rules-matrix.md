# 异常订单最终主判规则表

日期：2026-03-27

## 1. 文档定位

本文不是设计讨论稿，而是异常订单全自动主判的规则契约表。

它的作用只有三个：

1. 给后续数据模型 V2 提供稳定输入。
2. 给 API、logic、worker、recovery 的代码改造提供统一标准。
3. 给运营、申诉、法务、风控说明“系统为什么这样判”。

## 2. 当前正式支持的 claim 类型

当前提交索赔入口真实支持的 claim_type 只有三种，定义见 [locallife/api/risk_management.go](locallife/api/risk_management.go#L112)：

1. `foreign-object`
2. `damage`
3. `timeout`

另外：

1. `food-safety` 在算法层存在，但已明确走独立食安链路，不进入异常订单主判，见 [locallife/algorithm/claim_types.go](locallife/algorithm/claim_types.go#L14)。
2. 历史 migration 中出现过 `delay`、`quality`、`missing-item`、`other` 等旧枚举，但不属于当前提交入口的正式主判范围。

因此，本规则表只覆盖：

1. `foreign-object`
2. `damage`
3. `timeout`

## 3. 主判总原则

### 3.1 平台先赔，责任后定

主判默认沿用现有实现和现有协议：

1. 用户先获得自动赔付结果。
2. 平台作为先行赔付方出款。
3. 如果责任明确，再向责任方追偿。
4. 如果责任不够明确，则平台兜底，不强行追责。

这与现有协议文本一致，见 [locallife/db/migration/000135_create_agreements.up.sql](locallife/db/migration/000135_create_agreements.up.sql#L86)。

### 3.2 责任交割边界

本规则表采用以下强约束：

1. 商户责任域：只覆盖菜品本身问题，包括食安、异物、餐品本体质量缺陷、出餐时即存在的错误。
2. 骑手责任域：骑手在取餐时有检查餐品完整性和包装完好的义务。
3. 一旦骑手完成取餐确认并带走餐品，非食品本身问题即进入骑手责任域。
4. 取餐后至送达前的遗漏、遗撒、丢失、破损、运输污染，统一归骑手责任域。

### 3.3 主判正式输出

当前版本只允许四种正式结果：

1. `merchant_recovery`
2. `rider_recovery`
3. `platform_fallback`
4. `user_restricted`

说明：

1. 不允许商户/骑手分摊作为当前外卖主判的正式模式。
2. `split` 只保留为未来扩展值，不参与当前生产主判。

## 4. 主判优先级

系统必须按以下优先级决策，不能跳步：

1. 资格校验：订单是否允许索赔、用户是否有资格提交。
2. 用户恶意确认：如果用户恶意分达到确认级，优先进入 `user_restricted`。
3. claim 类型责任域路由：确定该 claim 属于商户责任域还是骑手责任域。
4. 责任域证据判定：在目标责任域内判断是否达到可追偿置信度。
5. 平台兜底：若未达到可追偿置信度，则进入 `platform_fallback`。

优先级禁止事项：

1. 不能因为服务方责任分略高，就跳过用户恶意确认。
2. 不能因为 claim_type 是 `damage` 或 `timeout`，就在证据缺失时自动追骑手。
3. 不能因为 claim_type 是 `foreign-object`，就在证据缺失时自动追商户。

## 5. 四类主判输入分数

主判读取四类分数：

1. `U`：用户恶意分
2. `M`：商户责任分
3. `R`：骑手责任分
4. `C`：证据置信分

本规则表只定义这些分数如何被使用，不在此重复展开建模细节。

## 6. 最终主判规则矩阵

### 6.1 claim 类型到责任域映射

| claim_type | 责任域 | 默认责任方 | 是否允许跨域回判 | 说明 |
| --- | --- | --- | --- | --- |
| `foreign-object` | 商户责任域 | 商户 | 不允许 | 只处理菜品本体问题，异物和食安同源 |
| `damage` | 骑手责任域 | 骑手 | 不允许 | 取餐后遗漏、遗撒、破损、丢失都归骑手 |
| `timeout` | 骑手责任域 | 骑手 | 不允许 | 履约时效问题归骑手 |

说明：

1. 当前版本不存在 `damage` 判商户，或 `foreign-object` 判骑手的正式路径。
2. 若未来业务要放开跨域回判，必须先修改责任交割规则，再改主判。

### 6.2 正式主判决策表

| 决策结果 | 触发条件 | 用户赔付 | 是否追偿 | 追偿对象 | 备注 |
| --- | --- | --- | --- | --- | --- |
| `merchant_recovery` | `claim_type=foreign-object` 且 `U < 80` 且 `M >= 70` 且 `C >= 60` | 是 | 是 | 商户 | 菜品本体问题明确 |
| `rider_recovery` | `claim_type=damage/timeout` 且 `U < 80` 且 `R >= 70` 且 `C >= 60` | 是 | 是 | 骑手 | 取餐后问题或履约时效问题明确 |
| `platform_fallback` | `U < 80` 且未达到对应责任域的追偿置信度 | 是 | 否 | 无 | 平台兜底，不强行追责 |
| `user_restricted` | `U >= 80` 且 `U >= M + 20` 且 `U >= R + 20` 且 `C >= 70` | 否或限赔 | 否 | 无 | 进入恶意限制路径 |

### 6.3 平台兜底进入条件

平台兜底不是异常分支，而是正式主判模式。只要出现下列任一类情况，就应优先进入 `platform_fallback`：

1. 目标责任域分数不足：
   `foreign-object` 下 `M < 70`；
   `damage/timeout` 下 `R < 70`。
2. 证据不足：`C < 60`。
3. 关键事实缺失：缺取餐确认、缺配送状态、缺有效异常摘要、缺核心设备/地址摘要。
4. 用户恶意分未到确认级，但服务方责任也未达到高置信度。
5. 模型输入冲突：同一单同时出现明显矛盾信号，无法稳定归责。

平台兜底禁止事项：

1. 不创建商户或骑手 recovery。
2. 不增加商户或骑手净责任计数。
3. 只记录平台兜底样本，用于后续校准和再学习。

### 6.4 用户恶意确认规则

用户进入 `user_restricted` 必须同时满足：

1. `U >= 80`
2. `U >= M + 20`
3. `U >= R + 20`
4. `C >= 70`

除此以外，还必须至少命中以下强确认条件之一：

1. 强图谱命中 + 高频索赔 + 高翻盘率。
2. 已确认团伙再次命中。
3. 多时间窗重复出现高恶意评分。

禁止直接定恶意的单一信号：

1. 同设备多人。
2. 同地址多人。
3. 近 7 天索赔次数高。
4. 首单即索赔。
5. 同一商户集中索赔。

这些信号只能贡献风险分，不能直接触发处罚。

## 7. claim_type 逐项规则

### 7.1 foreign-object

#### 正式口径

1. 责任域：商户责任域。
2. 主判结果只可能是：`merchant_recovery`、`platform_fallback`、`user_restricted`。
3. 不允许判骑手，也不允许商户/骑手分摊。

#### 进入商户追偿的条件

1. `M >= 70`
2. `C >= 60`
3. `U < 80`
4. 最好同时满足至少一个增强条件：
   独立用户同类投诉升高；
   商户同类异常率高于区域基线；
   历史同类申诉失败率高。

#### 进入平台兜底的条件

1. `M < 70`
2. 或 `C < 60`
3. 或关键商户异常画像尚未稳定

#### 禁止事项

1. 不因取餐后链路异常把 `foreign-object` 责任转给骑手。
2. 不因用户 claim 次数高，就在商户责任明确时免除商户责任；此时应由恶意规则单独决策。

### 7.2 damage

#### 正式口径

1. 责任域：骑手责任域。
2. 主判结果只可能是：`rider_recovery`、`platform_fallback`、`user_restricted`。
3. 不允许判商户，也不允许商户/骑手分摊。

#### 进入骑手追偿的条件

1. `R >= 70`
2. `C >= 60`
3. `U < 80`
4. 必须满足取餐后责任链成立：订单存在骑手履约链路，且能确认问题发生在取餐后责任域内。

#### 进入平台兜底的条件

1. `R < 70`
2. 或 `C < 60`
3. 或取餐确认、配送事实、异常画像缺失导致无法高置信追骑手。

#### 禁止事项

1. 不能因为餐损证据弱，就直接转判商户。
2. 不能因为骑手历史画像差，就在当前单证据严重缺失时直接追骑手；此时应进入平台兜底。

### 7.3 timeout

#### 正式口径

1. 责任域：骑手责任域。
2. 主判结果只可能是：`rider_recovery`、`platform_fallback`、`user_restricted`。
3. 不允许判商户，也不允许商户/骑手分摊。

#### 进入骑手追偿的条件

1. `R >= 70`
2. `C >= 60`
3. `U < 80`
4. 存在有效履约时效异常事实，例如严重超承诺时间或明显脱离区域同时间段基线。

#### 进入平台兜底的条件

1. `R < 70`
2. 或 `C < 60`
3. 或天气、运力、状态链路异常导致无法稳定归责。

#### 禁止事项

1. 不能仅凭 claim_type 为 `timeout` 就自动追骑手。
2. 不能在状态链不完整、回调滞后、系统时钟异常等情况下直接追骑手。

## 8. 主判落库要求

每次裁决至少必须落以下字段：

1. `decision_mode`
2. `claim_type`
3. `responsibility_domain`
4. `user_risk_score`
5. `merchant_liability_score`
6. `rider_liability_score`
7. `confidence_score`
8. `score_breakdown`
9. `graph_hits`
10. `fallback_reason`，仅平台兜底时必填
11. `restriction_reason`，仅用户限制时必填

主判快照必须能回答三个问题：

1. 为什么赔了用户。
2. 为什么追了某个责任方，或者为什么没追。
3. 为什么没有把这单判成恶意。

## 9. 画像更新规则

### 9.1 允许增加画像净值的场景

1. `merchant_recovery`：增加商户净责任计数。
2. `rider_recovery`：增加骑手净责任计数。
3. `user_restricted`：增加用户确认恶意计数。

### 9.2 禁止增加画像净值的场景

1. `platform_fallback`
2. 证据不足的观察态
3. 已被申诉再裁决撤销的历史判定

### 9.3 申诉再裁决回滚规则

如果后续申诉成功或再裁决推翻原判，则必须：

1. 回滚对应净责任计数。
2. 回滚用户恶意净值。
3. 回滚 recovery 或限制动作。
4. 保留原始事件，但标记为 overturned。

## 10. 与现有实现的差异

当前实现和本规则表相比，至少有四个明显差异：

1. 现有实现更像 `claim_type -> 默认责任方`，而本规则表要求 `claim_type -> 责任域 -> 置信追责或平台兜底`。
2. 现有实现在行为异常时会走 `platform-pay`，但还没有把平台兜底定义成正式主判模式。
3. 现有实现对用户恶意判断过度依赖近 90 天 claim 数，本规则表要求使用净有效口径。
4. 现有实现没有把“禁止信号”和“必须条件”分开，本规则表明确了哪些信号只能加分不能定罪。

## 11. 后续阶段的直接输入

本规则表完成后，下一阶段数据模型 V2 需要直接服务于以下目标：

1. 支持 `decision_mode` 四种正式结果。
2. 支持平台兜底理由落库。
3. 支持用户恶意、商户责任、骑手责任、置信度四类分数字段。
4. 支持净有效画像和 overturned 事件回滚。

如果后续数据模型或代码实现与本规则表冲突，应以本规则表为准先修正文档或实现，不允许两者长期漂移。
