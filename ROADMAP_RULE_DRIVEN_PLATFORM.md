# 规则驱动本地餐饮基础设施落地计划（高层规划）

> 目标：在现有代码基础上，统一抽象、模块化规划，实现堂食扫码点餐/外卖/预订三端闭环，多区域运营隔离（按区县 region_id），平台收付通实时分账，以及异常订单规则驱动裁决的可配置化。

## 1. 目标与边界
- 目标业务链路：顾客端（小程序）、商户端（Web）、运营商端、平台端全链路闭环。
- 多租户原则：以区县 region_id 为核心隔离域（一个商户必属于一个区县）。
- 规则优先：所有可规则化场景避免人工干预，例外场景必须可追溯。
- 一次维护，三端销售：同一业务能力在三端复用同一后端能力与规则。

## 2. 统一抽象与全局设计
### 2.1 统一域模型
- 订单域：订单主状态 + 履约状态 + 支付状态 + 分账状态（统一时序与约束）。
- 履约域：堂食/外卖/预订共用履约引擎（同一事件驱动接口）。
- 规则域：统一规则定义、版本、发布、灰度、回滚、审计。
- 租户域：区县为租户边界（region_id），统一中间件/查询约束。

### 2.2 统一规则引擎抽象
- 规则对象：
  - 适用范围（订单类型、区域、商户、用户角色）
  - 触发条件（状态、金额、时间、行为）
  - 动作（放行/拒绝/改价/补偿/转人工/告警）
  - 优先级、版本、灰度策略
- 规则执行：事件驱动 + 事务内校验 + 异步补偿。

### 2.3 统一可观测与审计
- 全链路审计日志：规则命中、状态流转、分账决策、异常裁决。
- 运营可视化：统一看板 + 可追溯证据。

## 3. 现有能力融入（高层复用策略）
- 风控与索赔：现有索赔自动审核 + 行为追溯合入统一规则引擎作为“风控规则集”。
- 分账：现有 profit_sharing_orders 作为分账执行与对账核心表，补齐规则配置与补偿流程。
- 运营商规则：现有 operator_rules 提供规则读取入口，升级为可配置规则源。
- 预订/堂食/外卖：统一状态机矩阵与事件触发点，复用同一履约/规则执行路径。

## 4. 分项落地计划（阶段化）

### 阶段 0：基线梳理与规范化（1-2 周）
- 统一状态机与错误码规范（API 对齐）。
- 统一租户边界约束策略（region_id 作为强约束）。
- 输出全链路 API/表/页面映射清单。
- 产出上线验收清单（性能、安全、合规、监控）。

**交付物**
- 状态机矩阵与规则约束文档
- 租户隔离规范
- API/表/页面映射清单
- 上线验收 checklist

### 阶段 1：规则引擎与配置中心（2-4 周）
- 规则存储：规则表设计（版本、灰度、范围）。
- 规则执行引擎：统一校验入口（订单、预订、分账、索赔）。
- 规则审计：命中日志、结果回放。
- 运营配置 UI（运营商/平台端）。

**交付物**
- 规则引擎框架
- 规则配置与发布流程
- 规则命中审计与回放

### 阶段 2：支付分账闭环（2-3 周）
- 分账策略配置化（平台/运营商/商户/骑手）。
- 分账失败补偿与重试流程。
- 实时分账 SLA 与对账核验。

**交付物**
- 可配置分账规则
- 分账重试与补偿机制
- 对账与差错处理流程

### 阶段 3：异常订单裁决闭环（2-3 周）
- 索赔/异常规则统一到规则引擎。
- 证据与行为追溯自动化。
- 人工介入仅保留“审计复核”，不进入主流程。

**交付物**
- 异常裁决规则集
- 证据链与追溯流水
- 可解释裁决报告

### 阶段 4：运营/平台控制台（2-4 周）
- 运营商端：规则配置、商户审核、统计与分账复核。
- 平台端：全局风控、分账对账、跨区县运营监控。

**交付物**
- 运营商与平台控制台
- 规则配置与生效可视化

## 5. 验收与上线标准
- 业务闭环：三端业务链路在同一规则引擎下可追溯闭环。
- 分账闭环：成功率、时效、失败补偿全覆盖。
- 异常裁决：95% 以上自动裁决覆盖率，人工介入仅复核。
- 多区县隔离：全链路 region_id 强约束且可审计。
- 可观测性：错误码标准化、告警/指标/审计日志完整。

## 6. 风险与依赖
- 规则引擎设计与灰度发布复杂度较高。
- 现有硬编码规则迁移到规则引擎需阶段性兼容。
- 分账补偿逻辑需与支付回调强一致性设计。

## 7. 下一步决策点
- 确认规则引擎抽象与最小可用能力（MVP）。
- 确认运营/平台端的功能边界。
- 确认分账 SLA 与补偿策略。
